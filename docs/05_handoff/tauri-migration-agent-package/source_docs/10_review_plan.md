## 3. 方針転換

### 本体価値の再定義

Auto Mosaic の本体価値は「特定モデルで特定対象を検出するツール」ではなく、**公開モデルを差し替えて使える汎用モザイク編集ソフト** に置く。

- 価値の中心は検出モデルそのものではなく、**編集ワークフロー・トラック運用・マスク修正・書き出し** にある
- 一般用途（顔モザイク・ナンバープレート・機密情報等）にも説明できる構成へ寄せる
- モデル依存 UI（ラベル表示・検出対象の制御）が今後重要になる。固定モデル前提の UI 設計を段階的に脱却する必要がある

### なぜこの転換が必要か

- 課金審査・決済プロバイダの審査で「アダルト専用ツール」扱いになると通過リスクが上がる
- LP・広告で説明しづらい製品は、そもそも人に見せられない
- レビュー者に渡すとき「普通のモザイク編集ソフト」として説明が成立しなければ、フィードバックの質も下がる

## 4. レビューにおける評価軸

ピンキーレビューで重視する観点を以下に整理する。レビュー者への依頼時にもこの軸を共有する。

| # | 観点 | 具体的に見ること |
|---|------|-----------------|
| 1 | 触ってすぐ分かるか | 動画を開いて検出→編集→書き出しの流れが、説明なしでも辿れるか |
| 2 | 編集の手触りが良いか | マスクの移動・リサイズ・頂点編集で「何を触っているか」が常に分かるか。反応にもたつきがないか |
| 3 | 複数トラックでも破綻しないか | 5〜10 トラック程度で、タイムライン・プレビュー・操作が混乱しないか |
| 4 | 一部だけモザイク除外できるか | 特定人物・特定対象だけモザイクを外す操作が直感的に行えるか |
| 5 | 現実の素材で使えるか | テスト動画ではなく、実際の撮影素材で検出→編集→書き出しが完走するか |
| 6 | 「普通のモザイク編集ソフト」として説明が成立するか | UI・ラベル・導線に、特定用途を前提とした表現が露出していないか |

## 5. 最優先タスク一覧

### A. タイムライン縦スクロール対応

**課題**: マスクトラックが増えたとき（5 本以上）、下の行が隠れて操作できなくなる。レビューで複数トラックを扱う場面で体験が破綻する。

| 項目 | 内容 |
|------|------|
| 行領域に縦スクロールを持たせる | トラックレーン部分を `QScrollArea` 等でスクロール可能にする |
| 選択行 auto-scroll | トラックを選択したとき、その行が見切れていたら自動スクロールで可視領域に入れる |
| 追加直後の行が見切れない | 新規トラック追加時に、追加された行が表示範囲に入るようにする |
| ルーラー / 時間軸は固定 | 縦スクロールしても、上部のルーラー（時間軸）はスクロールしない。横スクロールとは独立に動作する |

### B. 特定トラックだけモザイク除外

**課題**: 現状の「表示 ON/OFF」はプレビューの可視性制御であり、書き出し対象の制御とは分離されていない。「この人だけモザイクを外す」が直感的にできない。

| 項目 | 内容 |
|------|------|
| 「表示 ON/OFF」と「書き出し対象 ON/OFF」を分離 | visible（プレビュー表示）と export_enabled（書き出し対象）を別フラグにする |
| TrackList への反映 | トラック一覧で書き出し対象状態が視覚的に分かるようにする |
| Inspector への反映 | プロパティパネルで書き出し対象のトグルを操作できるようにする |
| Export 反映 | ExportService が export_enabled=False のトラックをスキップする |
| Undo 対応 | 書き出し対象の切り替えが Undo/Redo の対象になる |

### C. 編集手触り・編集リアクション改善

**課題**: 現状でも編集は動作するが、「どのトラックを触っているか」「操作が反映されたか」の即時フィードバックが弱い。レビュー者の第一印象に直結する。

| 項目 | 内容 |
|------|------|
| 選択の分かりやすさ | 選択中トラックのハイライト強化（プレビュー上のバウンディング表示、トラック一覧の背景色） |
| 反応速度の体感 | フレーム移動時のプレビュー更新、マスク描画の遅延低減 |
| 操作対象の明確化 | プレビュー上でホバー中のトラックが分かるフィードバック（カーソル変化、薄いハイライト等） |
| キーフレーム編集時の反応 | KF 追加・削除時にタイムラインマーカーが即座に更新される |
| トラック切替時の反応 | トラック一覧で切り替えたとき、プレビューの選択マスクが即座に追従する |

### D. 後ろ向き追跡補完の整理

**課題**: 動画の途中から検出されたトラックを、検出開始より前のフレームに向かって補完したい。ユーザー要望として価値が高いが、実装は重い。

レビュー前の判断材料として、2段階に分けて整理する。

| フェーズ | 内容 | 5月レビュー判断 |
|---------|------|----------------|
| Phase 1: UI / レビュー導線 | 「このトラックの開始フレームより前に遡って補完したい」という操作の入口を用意する。実際の補完はしないが、導線・UIフローとして存在させる | **✅ 実装済み** |
| Phase 2: 実際の過去方向補完 | backward tracking / backward propagation の実装。直前のキーフレーム形状を起点に、逆方向にフレームを走査して位置を推定する | レビュー後 |

**判断根拠**: Phase 2 は検出パイプラインの拡張を伴い、品質検証にも時間がかかる。レビュー前に半端な実装を入れると、バグや精度問題でレビュー体験を損なうリスクがある。Phase 1 で「やろうとしていること」をレビュー者に伝えるだけでも価値がある。

### E. モデル差し替え前提の UI 基盤

**課題**: 現状の UI は NudeNet / EraX の固定ラベルを前提にしている。汎用モザイク編集ソフトとして見せるなら、エンジン選択 UI とラベル制御の基盤が必要。また、検出モデルは原則同梱しない方針に変更されたため、ユーザーが選択・切替できる導線と、将来のモデル管理UIへの接続口も合わせて整備する。

| 項目 | 内容 |
|------|------|
| 検出エンジン選択（ツールバー） | ツールバーにコンボボックスを追加。現在の3バックエンドをユーザー向け名称で表示する |
| 現在のエンジン常時表示 | ステータスバーに「使用中の検出エンジン名」を常時表示 |
| ラベル一覧の動的表示 | `get_label_list()` インターフェース追加。選択エンジンに応じてカテゴリ一覧を動的表示 |
| ラベルフィルタの最小構成 | 「この対象カテゴリだけ検出する」「この対象カテゴリは無視する」の基本制御 |
| ラベルが多い場合の UI 整理 | スクロール・折りたたみ等で、カテゴリ数が多いモデルでも破綻しない |
| 詳細設定導線 | 「詳細設定...」リンクで既存の GPU 設定ダイアログへ接続 |
| 将来の管理UIへの接続 | コンボボックス末尾に「エンジンを管理...」枠を確保（Phase 2 で実装）|

**5月レビュー前の線引き**: エンジン選択 UI・ラベル一覧・最小フィルタまで。エンジン管理ダイアログ（モデル追加・詳細表示等）は Phase 2。

### F. ピンキーレビューの渡し方

**課題**: ソフトを渡すだけでは、レビュー者は何を見ればいいか分からない。渡し方自体を設計する必要がある。

| 項目 | 内容 |
|------|------|
| レビュー用ビルド | `setup.bat` + `run_ui.bat` で起動できる状態を検証。非開発者が迷わない手順書を添える |
| サンプル動画 | レビュー用に 1〜2 本の短尺サンプル動画を用意する。一般素材（顔モザイク等）を使う |
| 触ってほしい順番 | 推奨操作フロー（開く→検出→編集→書き出し→除外→Undo）を簡潔にまとめる |
| フィードバック観点 | セクション 4 の評価軸をレビュー者向けに書き直したチェックリスト |
| 既知制限の共有 | レビュー時点での既知の制限・未実装をあらかじめ伝え、「壊れている」と「まだ作っていない」を区別してもらう |

#### モデル非同梱前提のレビュー手順

配布物に検出モデルを同梱しない方針のため、レビュー者はレビュー開始前にモデルを自分で追加する必要がある。手順書にこの工程を含めること。

**モデル追加の選択肢（手順書に記載する）**

| 方法 | 概要 | 難易度 |
|------|------|:-----:|
| setup.bat 実行時に自動取得 | `setup.bat` が NudeNet の ONNX モデルを自動ダウンロードする。インターネット接続があれば最も簡単 | 低 |
| 手動配置 + 再スキャン | モデルファイルを `models/` に手動配置し、アプリ起動後に再スキャンを実行 | 中 |

> **注**: 5月レビュー時点の MR-05 実装（エンジン選択コンボ）では、バックエンドとして NudeNet 320n が引き続き動作する。手順書では「setup.bat を実行するとモデルが取得される」フローを案内するのが最もシンプル。

**モデル未追加時にできること / できないこと**

| 操作 | モデルなし | 備考 |
|------|:--------:|------|
| 動画を開く / タイムライン操作 | ✅ | モデル不要 |
| 既存プロジェクトの読み込み・編集 | ✅ | 保存済みマスクを使って編集継続可 |
| マスクの手動追加・編集 | ✅ | 検出なしで手動トラックを作れる |
| 書き出し | ✅ | 既存トラックがあれば可 |
| AI 検出の実行 | ❌ | モデルファイルが必要 |
| ラベル一覧の表示 | ❌ | アダプタのロードが必要 |

手順書には「モデル未導入でもエディタとして一部操作できる」ことを記載し、レビュー者が検出を試みる前にモデル導入を完了しているかを確認できるようにすること。

## 9. リスクと注意点

### 見せ方のズレ

「普通のモザイク編集ソフト」として見せたいが、内部実装のラベル名・UI 文言・検出設定にアダルト用途を前提とした表現が残っていると、レビュー者の印象がズレる。レビュー前に UI 上の文言を棚卸しする必要がある。

### モデル差し替え前提と UI 設計の整合

モデル差し替え前提にするなら、UI 設計を固定モデル前提のままにできない。MR-05（エンジン選択 UI + ラベル一覧 + 最小フィルタ）は最低限だが、検出設定ダイアログや閾値スライダー等にもモデル依存の要素が残っている可能性がある。また、UI 上に内部モデル名（「NudeNet」「EraX」「genital_m」等）が露出していないかの文言棚卸しも MR-05 の範囲として実施すること。

### 教師データとモデル差し替えの矛盾（2026-04-13 追記）

**問題**: モデル差し替えを前提にすると、異なるモデルが異なる用途のマスクを生成する。ユーザーが顔モデルで編集した結果とアダルトモデルで編集した結果を、区別せずに教師データとして蓄積すると以下の問題が起きる。

1. **ドメイン混在**: 顔検出モデルのファインチューニングにアダルトデータが混入する（逆も同様）。モデルの精度が下がる
2. **データの性質**: 用途によっては保存すべきでないデータが教師データに含まれる
3. **ユーザーの意図**: 「このモデルの学習に使ってほしい」という意図と「全モデルに使ってほしい」は異なる

**なぜ今整理すべきか**: ローカル学習はこの製品の最大の差別化ポイントであり、モデル差し替え前提と教師データ蓄積は矛盾しうる。設計を間違えると「使えない学習機能」か「使えない汎用性」のどちらかになる。

**設計原則（確定）**:

以下の 4 原則を `02_future-retraining-memo.md` に正式記録した。P4 実装における不変前提として扱う。

1. 差し替えモデルは **detect/edit 用** と **detect/edit/train 用** を分けて扱う
2. 教師データは **モデル系列ごとに分離保存** し、混合学習は行わない
3. unknown / imported custom model では教師データ保存を **自動有効化しない**
4. 学習対応は **明示的に許可したモデルファミリだけ** に限定する

これらの原則に基づく具体的な実装設計は、5月レビュー後の最初の設計議論で扱う。

**詳細**: [`02_future-retraining-memo.md`](02_future-retraining-memo.md) の「モデル差し替え前提での教師データ管理原則」セクションを参照。

### 後ろ向き追跡の実装コスト

後ろ向き追跡補完は「あれば大きな差別化」だが、実装は検出パイプラインの双方向化を伴う。Phase 1（UI 導線のみ）に留める判断を守り、Phase 2 をレビュー前に無理に入れない。

### ~~タイムライン縦スクロールの影響範囲~~ ✅ 解消

MR-01 完了時にテストで確認済み。ルーラー固定・横スクロール・ズームの既存動作に問題がないことを確認した。
