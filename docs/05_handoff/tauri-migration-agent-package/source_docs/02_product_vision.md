# プロダクト要件定義書（完成像）

最終更新: 2026-04-11

## この文書の位置づけ

この文書は、Auto Mosaic という製品が「完成した」と言える状態を、実装に依存しない形で記述したものです。

実装の詳細（採用フレームワーク、クラス構成、アルゴリズム、設定値、ファイル配置、特定モデル名など）は意図的に書きません。理由は、実装詳細を要件として固定してしまうと、別の技術スタックで再実装する際に「現状の写し」になってしまい、本来達成したかったゴールから外れる危険があるからです。

この文書を読む人は、次のいずれかを行えるはずです。

- 製品が「何を実現すべきか」を、現行実装と切り離して把握する
- 別の技術スタックでゼロから組み直しても、本質的に同じ価値の製品を作る
- 機能追加・改修案を「完成像に貢献するか」で判断する

現行の実装ベースの詳細、現時点で達成済みの範囲、達成方法については、別文書（`02_design/` 配下、`03_planning/01_remaining-tasks.md` 等）を参照してください。

## 製品が解決する問題

想定ユーザーは次の状況に置かれています。

- 手元にあるローカル動画の中の特定領域を、視聴可能な状態を保ったまま見せたくない
- 動画を外部サービスに送信することはしたくない / できない
- 全フレームを手作業で塗るのは現実的ではない
- AI に完全自動で任せると、誤検出・見落としの責任を負わされてしまう
- 編集を中断して、後から再開したい
- 仕上げた成果物を 1 本の動画ファイルとして取り出したい

この製品の存在理由は、これらすべてを 1 つのデスクトップアプリの中で完結させることです。

## プロダクトの核となる価値

1. **ローカル完結**: 動画データは一切アプリ外に出さない
2. **AI 補助 + 人間が最終決定**: AI は候補を出す役割であり、最終判断と修正権限は常にユーザーに残る
3. **断片ではなくトラックで考える**: 同じ被写体は時間を通じて 1 本の編集対象として扱える
4. **修正は要所だけで済む**: 全フレームを触る必要はなく、要所だけ直せば残りは自動で埋まる
5. **長尺でも壊れない**: 長い動画でも、編集 / 保存 / 書き出しのどこかで破綻しない

これらは「あれば嬉しい」ではなく、「ひとつでも欠けたら製品として成立しない」中核要求です。

## 想定ユーザーと使い方

### 想定ユーザー像

- 自分の PC に動画を持っていて、それを自分や限定的な相手に渡したい個人
- プレビュー操作・キーフレーム編集に慣れる程度の編集リテラシーがある
- 動画を外部に送りたくない、または送れない事情がある

### 想定しないユーザー像

- 動画を中央サーバーに送ってクラウド処理させたい人
- ブラウザだけで完結させたい人
- スマートフォン単独で操作したい人
- 複数人でリアルタイム共同編集したい人
- 全自動に任せて成果物だけ受け取りたい人

### 典型的な利用フロー

1. アプリを起動し、ローカル動画を読み込む
2. AI による検出を実行し、初期の候補を得る
3. 候補を眺めて、不要なものを消し、見落としを補う
4. 動きの激しい区間で要所のフレームを微調整する
5. 書き出し前に、リスクのありそうなフレームを確認する
6. 動画として書き出す
7. 途中で中断した場合は、次回起動時に作業を継続できる

このフローのどの段階で抜けても、ユーザーが致命的な損失を被らないことが製品の前提です。

## 機能要件（責務ベース）

実装方法には踏み込まず、「この機能が満たすべき責務」を記述します。

### 動画読み込み

- ユーザーがローカルの動画ファイルを開ける
- 開いた動画の長さ・解像度・フレームレートなどの基本情報がアプリ側に正しく伝わる
- 一般的な編集用途で扱う動画フォーマットを取り扱える
- 開けない動画については、理由がユーザーに分かる形で伝わる

### 自動検出（AI 補助）

- ユーザーが任意のタイミングで検出を実行できる
- 動画全体を対象にできる
- ユーザーが指定した特定区間だけを対象にできる
- 現在表示しているフレーム単発だけを対象にできる
- 検出の進捗がユーザーに見え、途中キャンセルできる
- 検出結果は、その後ユーザーが編集できる形でアプリ内に取り込まれる
- 検出のバックエンド（モデル）は将来差し替え可能であり、特定の実装に永続的に縛られない

### 編集対象（マスク）

- 1 つの被写体（追跡対象）は、時間を通じて 1 本の編集対象として扱える
- 一時的に検出が途切れても、編集対象は即座には消えない
- 編集対象には少なくとも次の情報が含まれる
  - 目に見える形（位置・大きさ・形）
  - 見せたくない領域への適用方法（例: モザイクの強度）
  - 時間的な存在範囲
- 1 つの動画に複数の編集対象が同時に存在できる
- 各編集対象は表示 / 非表示を切り替えられる

### マスク編集

- ユーザーはプレビュー上で位置・大きさを直接ドラッグして変更できる
- ユーザーは形を頂点単位で編集できる（追加・移動・削除）
- ユーザーは任意のフレームを「ここはこの形・この位置」と確定できる
- 確定済みのフレーム間は自動で補間される
- ユーザーが手で確定したフレームは、自動処理によって勝手に上書きされない
- 編集はフレーム断片の独立操作ではなく、編集対象の時間的連続性を保つ操作として成立する

### タイムライン

- 動画のどこに編集対象が存在しているかを俯瞰で確認できる
- 現在再生位置が常に視覚的に分かる
- 必要に応じて拡大・スクロールで細部を見られる
- ユーザーが「この区間だけ操作する」という指定ができる
- 編集対象ごとに、確定済み・補間中・予測中などの状態の違いが視覚的に区別できる

### 再生とプレビュー

- 動画を再生・停止できる
- フレーム単位で前後に移動できる
- モザイク適用後の見え方をプレビューで確認できる
- 再生中は誤操作で編集が壊れないように保護される

### 取り消し / やり直し

- ユーザーの編集操作は履歴として残る
- 直前の操作を取り消せる、取り消した操作をやり直せる
- ユーザーから見て「いま何の操作が取り消し対象か」が把握できる

### 書き出し

- 編集結果を 1 本の動画ファイルとして書き出せる
- プレビュー用の見た目ではなく、最終フレームに対してマスクが適用された結果が出力される
- 出力フォーマット・解像度・品質などの基本設定が選べる
- 元動画に音声がある場合は、原則として音声を維持できる
- 進捗が見え、ユーザーが途中でキャンセルできる
- 書き出しジョブを 1 本ずつ実行するだけでなく、複数まとめて順番に処理できる
- アプリを閉じて再起動しても、未完了の書き出しジョブが失われない

### 書き出し前の安全確認

- 書き出し実行前に、マスク欠落などのリスクが疑われる箇所をユーザーが確認できる
- ユーザーが内容を確認した上で、書き出しを進めるか判断できる
- 「書き出してから気付く」事態を避けるために存在する機能である

### プロジェクトの保存と再開

- 編集中の状態（読み込んだ動画への参照、編集対象、その時間的存在）を保存できる
- 保存したものを後から開いて編集を続けられる
- 未保存の編集状態はクラッシュに耐え、次回起動時に復元の機会が与えられる

### 設定と環境

- 推論や書き出しに使う計算リソース（GPU / CPU など）に関する設定が変更できる
- GPU が使えない環境では自動的に CPU で代替動作する
- 環境に依存する設定を変えても、ユーザーの編集データそのものは壊れない

## 非機能要件

### プライバシー / セキュリティ

- 動画データはユーザーの PC の外に出ない
- 推論・編集・書き出しのいずれの段階でも、外部送信を伴わずに完結する
- インターネット接続なしでも全機能が動作する

### 互換性

- Windows を主対象とするが、macOS / Linux でも動作可能な構成を維持する
- 特定の検出モデル・特定の GPU ベンダーに永続的に縛られない

### 安定性

- 長時間動画・数千フレーム規模の編集でも、編集 / 保存 / 書き出しのいずれかが破綻しない
- 編集途中の意図しない終了でも、ユーザーの作業成果が極力失われない

### パフォーマンス目標

- 編集中のプレビュー・フレーム移動はインタラクティブに感じられる速度である
- 自動検出と動画書き出しは、進捗が見え、かつキャンセルが効く
- 高速化に必要な計算リソース（GPU 等）を使える環境では、それを活かせる

### 操作性

- 主要操作はキーボードと、画面要素クリック / ドラッグの組み合わせで完結できる
- 編集対象の現在の状態（編集中・確定済み・補間中・予測中など）が視覚的に分かる
- ユーザーが「いまどの操作モードにいるか」が、画面のどこかに常に見える

### 復旧可能性

- 異常終了からの復旧導線がある
- 書き出し中の中断・再起動からも、未完了ジョブを再処理できる導線がある

## 設計上の不変原則

実装方法は問わないが、次の原則は崩してはならない。

1. **AI は補助、人間が最終決定者**  
   AI の出力を上書き不能な真実として扱わない。最終結果は常に人間が修正可能で、その修正は以後も保護される。

2. **時間軸を持つ編集対象**  
   編集対象はフレーム単位の独立物ではなく、時間を通じて連続する 1 本の存在として扱う。検出が一時的に切れただけで編集対象を消失させない。

3. **手修正の優先**  
   自動処理は、ユーザーが手で確定したフレームを上書きしてはならない。

4. **外部送信ゼロ**  
   ユーザーの動画データは、いかなる経路でもアプリ外に出てはならない。

5. **フォールバック必須**  
   望ましい計算リソース（GPU、上位モデル、高品質な輪郭抽出など）が使えない環境でも、ユーザーが目的を達成できる経路が必ず存在する。

6. **プレビューと書き出しの一致**  
   プレビューで見た結果と、書き出された動画フレームの結果は、原則として一致する。

7. **編集中の事故防止**  
   再生中・書き出し中など、ユーザーが意図しないタイミングでの編集事故を防ぐ仕組みがある。

## スコープ外

次の機能は本製品の完成像には含めない。

- クラウドへの動画アップロード
- 中央サーバーでの推論
- 複数人による同時編集
- モバイル単独での動作
- ブラウザ上での動作
- 教師データの自動回収・自動再配布

なお、ローカルでの教師データ蓄積・ローカル再学習は将来拡張のスコープであり、本書では取り扱わない。詳細は `01_requirements/03_p4-retraining-requirements.md` を参照。

## 完成と判断する条件

本製品が「完成した」と判断するには、次の全てを満たしていなければならない。

1. ユーザーは、ローカル動画を読み込んでから書き出すまでを、外部ガイドなしで一通り遂行できる
2. AI 検出は、ユーザーが「修正対象」として扱える形で結果を残す
3. ユーザーが手で行った修正は、後続の自動処理によって失われない
4. 編集対象は、検出が一時的に途切れても 1 本のまま維持される
5. 編集途中で異常終了しても、次回起動時に作業を続けられる
6. 書き出した動画は、プレビューで確認した内容と一致した結果を持つ
7. 書き出しは進捗が見え、キャンセルでき、複数ジョブを順に処理できる
8. GPU が無い環境でも全機能が CPU で動作する
9. 動画データは外部に送信されない
10. 操作性・視覚情報により、ユーザーが「いま何の状態か」を見失わない

これら 10 項目のいずれかが満たされない場合、本製品はまだ完成していないものとする。

## 改修・追加要望の扱い

新規機能や改修案がこの完成像に貢献するかどうかは、次の問いで判断する。

- それは「核となる価値」のいずれかを強化するか
- それは「設計上の不変原則」に反していないか
- それは「想定ユーザー像」が必要とするものか
- それを入れなかった場合、「完成と判断する条件」のいずれかを満たせなくなるか

すべての答えが「いいえ」であれば、その案はスコープ外である可能性が高い。
