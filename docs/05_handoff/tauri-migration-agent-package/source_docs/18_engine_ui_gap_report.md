# 検出エンジンUI — 新旧仕様ギャップ分析レポート

作成日: 2026-04-14
最終更新: 2026-04-14 — 本レポートで指摘した BUG-1 / 表示名問題は対応済み（表示名はモデル名、カテゴリフィルタはツールバーのみ、古い設定の互換性は `GpuConfig.load/save` で吸収）。

---

## 1. 現状の確認済みバグ

### BUG-1: 顔チェックボックスが外れない

**再現手順:**
1. NudeNet 使用中に「顔」カテゴリを有効にする
2. **GPU設定ダイアログ内で**エンジンを EraX に変更して OK
3. GPU設定ダイアログを再度開く
4. 「顔」チェックボックスが **チェック済み＋無効化（グレーアウト）** 状態

**根本原因 — 2つの欠陥の複合:**

1. **ダイアログ内のエンジン切替時にチェックボックスが更新されない**
   - `gpu_settings_dialog.py` の `_detector_backend` コンボボックスに `currentIndexChanged` シグナルが未接続
   - エンジンを変更しても、ラベルフィルタの有効/無効が連動しない
   - ツールバーのコンボには `_on_engine_combo_changed` → `_trim_unsupported_categories` が接続されているが、ダイアログ内は未対応

2. **`_apply_and_close` が無効化チェックボックスも保存する**
   - `checkbox.isChecked()` は disabled 状態でも True を返す
   - 結果: EraX が非対応の "face" が `enabled_label_categories` に残留
   - 次回ダイアログを開くと「チェック済み＋disabled」で詰む

**影響範囲:**
- `data/config/gpu_config.json` に不正な状態が永続化される
- 現在の設定: `"detector_backend": "erax_v1_1"` + `"enabled_label_categories": ["genital_f", "face"]`
- "face" は EraX の `supported_categories` に含まれないため、検出時に無視されるが、設定上の不整合

## 2. 新旧仕様の対比

### 旧仕様（GPU設定ダイアログ一括型）

```
メニュー: 設定 > 検出エンジン/デバイス設定...
  └── GpuSettingsDialog
        ├── デバイス状態（CUDA/CPU）
        ├── デバイス選択（auto/cuda/cpu）
        ├── 推論設定
        │     ├── 検出エンジン選択      ← ★ エンジン選択
        │     ├── サンプリング間隔
        │     ├── 推論解像度
        │     ├── バッチサイズ
        │     └── 信頼度閾値
        ├── 検出対象（ラベルフィルタ）  ← ★ カテゴリ選択
        ├── 輪郭抽出設定
        └── VRAM節約
```

- **導線**: メニュー 1か所から全設定を一括操作
- **問題点**: 頻繁に切り替えるエンジン・カテゴリに毎回ダイアログが必要

### 新仕様（ツールバー分離型）

```
ツールバー:
  ├── [検出エンジン: ▼ コンボ]      ← ★ エンジン選択（即時切替）
  ├── [対象カテゴリ ▾ メニュー]     ← ★ カテゴリ選択（即時保存）
  └── [詳細設定...]                  ← GPU設定ダイアログを開く

ステータスバー:
  └── "エンジン: 〇〇"               ← 現在のエンジン表示

メニュー: 設定 > 検出エンジン/デバイス設定...
  └── GpuSettingsDialog（旧仕様と同じ内容がそのまま残存）
```

- **導線**: ツールバーで即時操作 + ダイアログで詳細設定
- **問題点**: エンジン選択とカテゴリ選択が **2か所に重複**

---

## 3. 重複の詳細

| 操作 | ツールバー | GPU設定ダイアログ | 同期方法 |
|------|-----------|-----------------|---------|
| エンジン選択 | `_engine_combo` (即時反映) | `_detector_backend` コンボ (OK時反映) | ダイアログ → ツールバー: `_sync_engine_combo_from_config()` で同期 |
| カテゴリフィルタ | `_category_menu` (即時保存) | `_label_checkboxes` (OK時反映) | ダイアログ → ツールバー: `_sync_category_menu_to_engine()` で同期 |

**同期の不整合リスク:**
- ダイアログ内でエンジンを変更 → ラベルチェックボックス未連動（BUG-1 の原因）
- ダイアログでキャンセル → ツールバー側は変更なし（正常動作）
- ツールバーでカテゴリ変更 → ダイアログ開き直すまで反映されない（正常動作）

## 6. 導線の構造的問題

### 現在の導線マップ

```
ユーザーが「検出エンジンを変えたい」
  ├── ツールバーのコンボから変更            ← 即時反映、カテゴリも連動
  └── 設定 > 検出エンジン/デバイス設定...    ← ダイアログ内で変更、OK で反映
        └── ダイアログ内のエンジンコンボ      ← カテゴリ連動しない（バグ）

ユーザーが「検出対象を変えたい」
  ├── ツールバーの「対象カテゴリ ▾」         ← 即時保存
  └── 設定 > 検出エンジン/デバイス設定...    ← ダイアログ内で変更、OK で反映
        └── ダイアログ内のチェックボックス    ← エンジン変更時に連動しない（バグ）

ユーザーが「新しいモデルを追加したい」
  └── 導線なし（手動配置のみ）               ← UIから認識不能

ユーザーが「エンジンの詳細を知りたい」
  └── ステータスバーに「エンジン: 〇〇」     ← 表示名のみ、詳細なし
```

### 混乱の原因

1. **2つの入り口**: 同じ操作（エンジン選択・カテゴリ選択）が2か所にあり、挙動が異なる
2. **ダイアログの責務過多**: GPU/デバイス設定、エンジン選択、カテゴリ選択、輪郭設定、VRAM設定が1つのダイアログに混在
3. **即時反映 vs OK確定**: ツールバーは即時反映、ダイアログはOK確定。ユーザーの期待値が統一されない
4. **抽象的な表示名**: モデル実体が隠蔽されていて、何が動いているかわからない

---

## 7. 提案: 適切な導線構築

### 方針: ダイアログからエンジン選択・カテゴリ選択を除去

ツールバーを**唯一のエンジン選択・カテゴリ選択の入り口**とし、GPU設定ダイアログは**デバイス・推論パラメータ・輪郭設定に専念**させる。

### 変更案

#### A. GPU設定ダイアログの整理

```
GpuSettingsDialog（変更後）
  ├── デバイス状態
  ├── デバイス選択
  ├── 推論パラメータ
  │     ├── サンプリング間隔
  │     ├── 推論解像度
  │     ├── バッチサイズ
  │     └── 信頼度閾値
  ├── 輪郭抽出設定
  └── VRAM節約
```

**除去:**
- ~~検出エンジン選択コンボ~~ → ツールバーに一本化
- ~~検出対象チェックボックス~~ → ツールバーの「対象カテゴリ ▾」に一本化

#### B. 表示名のモデル名化

| backend ID | 変更後の display_name |
|-----------|---------------------|
| `nudenet` | NudeNet 320n |
| `nudenet_640m` | NudeNet 640m |
| `erax_v1_1` | EraX v1.1 |

ステータスバー表示例: `エンジン: NudeNet 320n`

#### C. BUG-1 の即時修正

重複除去とは別に、BUG-1 は以下で修正:

1. `gpu_settings_dialog.py` の `_apply_and_close` で disabled チェックボックスを除外:
   ```python
   self._gpu_config.enabled_label_categories = [
       cat_id for cat_id, checkbox in self._label_checkboxes.items()
       if checkbox.isChecked() and checkbox.isEnabled()
   ]
   ```

2. 不正状態の `gpu_config.json` をクリーンアップ（"face" を除去）
