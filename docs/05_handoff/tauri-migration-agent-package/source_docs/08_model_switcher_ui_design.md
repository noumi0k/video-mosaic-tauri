## 1. 問題設定

### 解くべき問題

Auto Mosaic を「NudeNet/EraX 固定の専用ツール」から「モデル差し替え可能な汎用モザイク編集ソフト」に転換するにあたり、**モデル選択という本質的に専門的な操作を、非エンジニアが安全かつ自然に行える UI** を設計する必要がある。

### 現状の課題

| 課題 | 現状 | 問題 |
|------|------|------|
| モデル選択の埋没 | GPU設定ダイアログ内の「Detector Backend」ドロップダウン | 技術者向け設定画面に埋もれており、製品のコア機能に見えない |
| 専門用語の露出 | "NudeNet", "EraX v1.1 (ONNX)", "Detector Backend" | 非エンジニアに意味が伝わらない |
| ラベルの固定性 | `label_schema.py` にアダルト用途のカテゴリがハードコード | 汎用ツールとしての見せ方と矛盾する |
| 拡張性の欠如 | `DetectorFactory` は3種固定、UIもハードコード | ユーザーがモデルを追加する導線がない |

### 設計で守るべき制約

1. **ユーザーはモデルに詳しくない** — 「ONNX」「推論」「バックエンド」を知らなくても選べる
2. **「何を使って検出しているか」は分かる必要がある** — ブラックボックスにはしない
3. **モデル数が増えても混乱しない** — 3個でも30個でも破綻しない構造
4. **自作・外部追加・標準を区別できる** — ただし区別が主張しすぎない
5. **将来の学習機能と整合する** — detect/edit 用と detect/edit/train 用の区分を受け入れられる
6. **学習機能の詳細は前面に出さない** — 今は「検出モデルを安全に選べる」ことが最優先
7. **アダルト専用ツールに見えない** — UI文言・ラベル・導線に特定用途の表現を出さない

## 2. 用語方針（全案共通）

| 内部概念 | UI 上の呼称 | 理由 |
|---------|------------|------|
| detector model / backend | **検出エンジン** | 「モデル」はML感が強い。「エンジン」は車・検索エンジンで馴染みがあり「中で動いている仕組み」を連想できる |
| label | **対象カテゴリ** | 「ラベル」はML用語。カテゴリは商品分類等で一般的 |
| confidence threshold | **検出の厳しさ** | 数値の意味が直感的に分かる |
| inference resolution | **検出解像度** | そのまま。ただし日常設定には出さない |
| batch size | 表示しない | 技術者向け。内部パラメータとして自動調整 |
| ONNX / CUDA / GPU | **詳細設定の中だけ**に表示 | 知っている人は探せばある。知らない人の目に触れない |
| fine-tuning / retraining | **学習** / **カスタム学習** | 「ファインチューニング」は通じない |
| detect/edit/train 用 | **学習対応** | 1語で済む |

## 3. UI 構成案

### 案A: ステータスバー起点 + フライアウトパネル

**コンセプト**: 現在使用中のエンジンをステータスバーに常時表示し、クリックでフライアウトパネルを開く。軽量で、既存レイアウトを壊さない。

**画面構成**:

```
┌─────────────────────────────────────────────────────────┐
│ [ツールバー]                                             │
├────────┬──────────────────────────┬──────────────────────┤
│        │                          │                      │
│ トラック│    プレビューキャンバス    │  プロパティパネル     │
│  一覧  │                          │                      │
│        │                          │                      │
├────────┴──────────────────────────┴──────────────────────┤
│ [タイムライン]                                           │
├─────────────────────────────────────────────────────────┤
│ ステータスバー   ...   [🔍 顔検出エンジン ▾]  F:120 ... │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼  クリックで開く
                    ┌─────────────────────┐
                    │ 検出エンジン         │
                    │ ───────────────────  │
                    │ ● 顔検出エンジン     │ ← 使用中
                    │   汎用・高速         │
                    │ ───────────────────  │
                    │ ○ 高精度エンジン     │
                    │   時間かかるが正確   │
                    │ ───────────────────  │
                    │ ○ [自作] マイモデル  │
                    │   4/10 作成          │
                    │ ───────────────────  │
                    │ [+ エンジンを追加]   │
                    │ [⚙ 詳細設定...]     │
                    └─────────────────────┘
```

**モデル一覧の見せ方**: 縦リスト。各項目にラジオボタン + 名前 + 1行説明。

**種別の区別**: 名前の前にバッジ（`[標準]` `[追加]` `[自作]`）。ただし標準はバッジなし（デフォルトだから）。

| 評価項目 | 評価 |
|---------|------|
| 非エンジニアへの分かりやすさ | ★★★★☆ — 常時見える＋フライアウトは軽量。ただしモデル数増加時に縦伸び |
| 将来の拡張性 | ★★★☆☆ — フライアウト内にタブ追加は可能だが窮屈。学習機能でモーダル昇格が必要になる可能性 |

**良い点**:
- 既存レイアウトへの影響が最小
- 「今何を使っているか」が常に見える
- 切り替えが1クリック+1クリックで完了
- モデルが2-3個の段階では最も軽快

**弱い点**:
- モデル数が10個を超えるとフライアウトが長くなりすぎる
- 各モデルの詳細情報（対応カテゴリ一覧など）を見せるスペースが足りない
- 将来の学習機能の入口を入れにくい

**向いているユーザー像**: モデルを頻繁に切り替えない。2-5個の中から選ぶだけ。設定を深く触らない。

**種別の区別**: セクションヘッダーで物理的に分離（「標準エンジン」「追加エンジン」「自作エンジン」）。

| 評価項目 | 評価 |
|---------|------|
| 非エンジニアへの分かりやすさ | ★★★★☆ — コンボボックスは馴染みのあるパターン。管理ダイアログはやや重いが日常は不要 |
| 将来の拡張性 | ★★★★★ — 右ペインに「学習」タブ追加、インポート/エクスポート等の余地が大きい |

**良い点**:
- 日常の切り替え（コンボボックス）と詳細管理（ダイアログ）が分離
- 既存の GPU 設定ダイアログの構造を活かせる（統合 or 並置）
- モデル数が増えてもカテゴリ分けで対応可能
- 将来の学習機能追加との接続が自然

**弱い点**:
- ツールバーの幅を消費する（現在のツールバーが既に整理されている）
- コンボボックスにモデル名が並ぶため、名前の付け方が重要になる
- 管理ダイアログは初心者にはやや重い印象を与える可能性

**向いているユーザー像**: モデルを何度か切り替えて試したい。自分でモデルを追加する可能性がある。ある程度の設定は自分でやりたい。

| 評価項目 | 評価 |
|---------|------|
| 非エンジニアへの分かりやすさ | ★★★★★ — 検出関連の全操作が1箇所。Premiere Pro 的なメンタルモデル |

- 現在の GPU 設定ダイアログの内容を自然に分解・再配置できる
- 「検出実行」ボタンがエンジン選択のすぐ近くにある

**弱い点**:
- プロパティパネルが縦に長くなりすぎる可能性（マスク設定との競合）
- 幅 260px ではカテゴリフィルタの表示が窮屈
- モデル管理（追加・削除・詳細）は結局別ダイアログが必要
- トラック選択中はマスク設定が優先されるため、表示切替が必要

**向いているユーザー像**: 検出→編集の一連の流れを1画面で完結させたい。設定を頻繁に触りながら試行錯誤する。

### 案D: メニュー起点 + 軽量選択モーダル

**コンセプト**: 日常的な切り替えは「検出」メニューまたはツールバーボタンから開く軽量モーダルで行う。管理機能は設定メニューから。

**画面構成**: メインウィンドウは変更なし。

**軽量選択モーダル**:


**モデル一覧の見せ方**: カード風の縦リスト。名前・1行説明・速度/精度の星評価。セクションヘッダーで分類。

**種別の区別**: セクション分け（「おすすめ」「追加したエンジン」「自作エンジン」）+ カード右上にバッジ。

| 評価項目 | 評価 |
|---------|------|
| 非エンジニアへの分かりやすさ | ★★★★★ — カード型で情報と視認性を両立。「おすすめ」で迷わない |

**良い点**:
- 初心者が「どれを選べばいいか」迷わない（おすすめセクション・星評価）
- カード型で情報量と視認性を両立
- メインウィンドウのレイアウトを一切変更しない
- 学習機能追加時にカードに「学習対応」バッジを追加するだけで済む

**弱い点**:
- モーダルを開く動作が1ステップ必要
- 「今使っているエンジン」がメインウィンドウ上で見えにくい（ステータスバー表示で補う必要あり）
- カテゴリフィルタ等の詳細設定をモーダル内に入れるとカード一覧と競合する

**向いているユーザー像**: 選ぶときにある程度の情報（説明・比較）がほしい。頻繁には切り替えないが、納得して選びたい。

## 4. 一覧項目の推奨

### 表示項目の評価

| 項目 | 一覧に常時表示 | 詳細ペインのみ | 非表示推奨 | 理由 |
|------|:-----------:|:----------:|:--------:|------|
| エンジン名 | ● | | | 最も基本的な識別子 |
| 1行説明 | ● | | | 何をするものか一目で分かる |
| 種別マーク | ● | | | 標準/追加/自作の区別（控えめに） |
| 使用中マーク | ● | | | 現在のアクティブエンジンが分かる |
| 検出対象の概要 | ● | | | 「顔」「人物」など1-2語で |
| 速度/精度の目安 | ● | | | ★ 形式で直感的比較 |
| 対応カテゴリ一覧 | | ● | | 情報量が多い。詳細に送る |
| 追加日時 | | ● | | 自作/追加のみ意味がある |
| メモ/説明文 | | ● | | ユーザーが任意に記入した補足 |
| 学習対応かどうか | | ● | | 将来機能。今は控えめに表示 |
| 推奨表示 | ● | | | 「おすすめ」バッジまたはセクション |
| ファイルパス | | | ● | 非エンジニアに不要 |
| ONNX バージョン | | | ● | 技術情報すぎる |
| ラベル数 | | ● | | 「対応カテゴリ: 6種類」程度で |

### 推奨構成

**一覧カード（4行以内）**:

```
[種別] エンジン名                    [使用中]
何を検出するか（1行）
速度: ★★★★ ・ 精度: ★★★
```

**詳細ペイン（展開時）**:

```
エンジン名
───────────────
種別: 標準（同梱）
検出対象: 顔（女性・男性）
対応カテゴリ: 6 種類
速度: ★★★★☆  精度: ★★★☆☆
───────────────
対象カテゴリ
☑ 顔（女性）  ☑ 顔（男性）  ☐ ...
───────────────
追加日: 2026-04-13
学習: 対応 / 非対応
メモ: （ユーザー記入欄）
```

### 項目数の原則

一覧の段階では **最大4行** に収める。5行以上になると、エンジンが5個並んだだけで画面が埋まる。詳細は「選択後に見る場所」を別に用意して逃がす。

## 5. ユーザーにどう選ばせるか

### 選択UIパターンの比較

| パターン | 少数時(2-3個) | 中規模(5-10個) | 多数(10+) | 初心者の迷い | 切替速度 |
|---------|:-----------:|:-----------:|:--------:|:----------:|:------:|
| ドロップダウン | ◎ | ○ | △ | 高い | 最速 |
| カード一覧（モーダル） | ○ | ◎ | ○ | 低い | 2手 |
| サイドバー常設 | △(空きすぎ) | ○ | ○ | 中 | 1手 |
| タブ切替 | △(タブ2個は不自然) | ○ | △(タブ多すぎ) | 中 | 1手 |
| ツールバーコンボ | ◎ | ○ | △ | 高い | 最速 |

### 推奨: 2段構成（日常選択 + 管理）

- **日常選択**: ツールバーまたはステータスバーのコンパクトな選択肢（コンボボックス or クリッカブルラベル）
- **管理・比較**: モーダルダイアログ（カード一覧 + 詳細ペイン）

この分離により:
- モデルが少ないうちは日常選択だけで十分回る
- モデルが増えたら管理ダイアログで比較・整理できる
- 初心者は管理ダイアログの「おすすめ」を見て安心して選べる
- パワーユーザーはコンボボックスで瞬時に切り替えられる

### 「どれを選べばいいか分からない」への対策

| 対策 | 方法 |
|------|------|
| おすすめ表示 | 標準エンジンに「おすすめ」バッジ。初期選択もこれ |
| 用途別ガイド | カードの1行説明を「〜を検出したいときに」形式に |
| 速度/精度の比較 | ★評価で一目で違いが分かる |
| デフォルト維持 | 初回起動時は標準エンジンが選択済み。変更しなくても使える |
| 詳細モード分離 | カテゴリフィルタ・閾値調整は「詳細設定」に分離し、基本画面では隠す |

## 6. 自作 / 外部追加 / 標準 の見せ分け方針

### 3つの種別定義

| 種別 | 定義 | 由来 | 学習対応 |
|------|------|------|---------|
| **標準** | アプリに同梱されているエンジン | 開発者が提供 | モデルによる |
| **追加** | ユーザーがファイルから取り込んだエンジン | 外部（由来不明の場合あり） | デフォルトOFF |
| **自作** | ローカル学習で作成されたエンジン | ユーザー自身の学習結果 | 対応（親モデルに準拠） |

### 一覧上での見分け方

**推奨: セクション分け + 控えめなバッジの併用**

```
── おすすめ ──────────────────────────
  顔検出エンジン                [使用中]
  高精度エンジン

── 追加したエンジン ──────────────────
  ナンバープレート検出      [追加]

── 自作エンジン ──────────────────────
  マイモデル v2             [自作]
```

**なぜこの方式か**:

- **セクション分け**: 視覚的に最も明確。スクロールしても「今どのカテゴリを見ているか」が分かる
- **バッジ**: セクションヘッダーが見切れている場合の補助。小さく控えめに
- **色分け**: やらない。色を増やすと UI が騒がしくなる。テーマカラー（青のアクセント）は「使用中」だけに使う
- **アイコン**: やらない。エンジンの種別を表す普遍的なアイコンがない。誤解を生むリスク

### 自作モデルの扱い

- 自作モデルは価値が高いが、一覧で目立たせすぎない
- 理由: 自作が1個で標準が2個のとき、自作が主張しすぎると「標準は劣る」印象を与える
- 自作エンジンのカードには、作成日と元モデル名を添えて由来を明示する
- 例: `マイモデル v2 — 顔検出エンジンをベースに学習（4/10 作成）`

### 外部モデルの扱い

- 外部から入れたモデルは由来が不明な場合がある
- 最低限: ファイル名・追加日を詳細ペインに表示
- ユーザー編集可能な「メモ」欄を設け、自分で由来を記録できるようにする
- 教師データ保存はデフォルトOFF（`02_future-retraining-memo.md` 設計原則3に準拠）

### 将来の拡張: 学習対応/非対応の区別

- セクションを「学習対応エンジン」「検出専用エンジン」に分ける **のではなく**、既存のセクション（標準/追加/自作）を維持しつつ、カードに「学習対応」バッジを追加する
- 理由: 学習対応/非対応はモデルの属性であり、分類軸ではない。分類軸を増やすとUIが複雑になる

## 7. 非エンジニア向けの情報設計

### 説明文のパターン

```
顔検出エンジン
→ 「人の顔を自動で見つけて追跡します」

高精度エンジン
→ 「時間はかかりますが、より正確に検出します」

ナンバープレート検出（追加）
→ 「車のナンバープレートを検出します」

マイモデル v2（自作）
→ 「顔検出エンジンをベースにカスタム学習したエンジンです」
```

### 詳細を隠しつつ必要な人には見せる構造

```
┌─ 基本設定（常に表示）─────────────────┐
│  検出エンジン: [顔検出エンジン ▾]      │
│  検出対象: 顔                         │
│  [▶ 検出する]                         │
└───────────────────────────────────────┘

▶ 詳細設定（折りたたみ、初期は閉じ）
┌───────────────────────────────────────┐
│  対象カテゴリ:                         │
│  ☑ 顔（女性）  ☑ 顔（男性）          │
│                                       │
│  検出の厳しさ: [====●======] 0.45     │
│  検出間隔: [5] フレームごと            │
│                                       │
│  ▶ デバイス設定（GPU/CPU）            │
│  ▶ 高度な設定                          │
└───────────────────────────────────────┘
```

**折りたたみは最大2段**。3段以上は非エンジニアが迷子になる。

### 初見ユーザーが怖くない UI にする工夫

1. **デフォルトが安全**: 何も変えなくても動く。標準エンジンが初期選択済み
2. **破壊的操作がない**: エンジンを切り替えても、既に作ったマスクは消えない
3. **元に戻せる**: 「おすすめに戻す」ボタン（またはデフォルト復元）
4. **説明が命令形でない**: 「〜してください」ではなく「〜できます」
5. **エラーが技術語でない**: 「ONNXセッション初期化失敗」→「このエンジンを読み込めませんでした。ファイルが正しいか確認してください」

## 8. 推奨案

### 結論: 案Bを軸に案A・Dの要素を取り込む

| 要素 | 採用元 | 内容 |
|------|--------|------|
| 日常の切替 | 案B | ツールバーにコンボボックス。エンジン名のみ。1手で切替完了 |
| 現在の状態表示 | 案A | ステータスバーにも現在のエンジン名を表示（常に見える） |
| 管理ダイアログ | 案B+D | カテゴリ分けセクション + カード風エントリ + 右ペイン詳細。案Dの「おすすめ」表示・星評価を採用 |
| 検出設定 | 案C 一部 | カテゴリフィルタ・閾値は管理ダイアログの詳細ペインに統合。プロパティパネルには入れない（幅の制約） |
| エンジン追加 | 案B | 管理ダイアログ下部にボタン |

### なぜこの案か

1. **日常操作と管理の分離がこの製品に合う**: 検出→編集→書き出しの主導線では、エンジンは「最初に選んで終わり」が多い。ツールバーのコンボボックスで十分。だが、モデルを比較したい・追加したいときには管理ダイアログが必要
2. **既存UIとの整合**: 現在の GPU 設定ダイアログは「デバイス設定（GPU/CPU）」と「検出設定（モデル・閾値・ラベル）」が混在している。これを分離することで、「デバイス設定」は技術者向けとして残し、「エンジン管理」は全ユーザー向けに新設できる
3. **5月レビューとの接続**: Phase 1 ではコンボボックス + ラベル一覧表示だけで済む（管理ダイアログは Phase 2 でよい）
4. **将来の学習機能との接続**: 管理ダイアログの右ペインに「学習」タブ/セクションを追加すれば、学習状態の表示・学習の開始がエンジン管理と同じ場所に収まる

## 9. Phase 1 / Phase 2 の切り分け

### Phase 1: 5月レビュー時点の最小構成

| 項目 | 内容 | 理由 |
|------|------|------|
| ツールバーにエンジン選択コンボボックス | 現在の3バックエンドを「検出エンジン」として表示。名前を人間向けに変換（NudeNet→顔検出エンジン等） | レビュー者に「モデル差し替え前提」が伝わる最小構成 |

| ステータスバーにエンジン名表示 | 現在のエンジン名を常時表示 | 「今何を使っているか」を常に確認できる |
| コンボボックスからGPU設定ダイアログへのリンク | 「詳細設定...」項目をコンボボックス末尾に追加 | 既存の設定画面への導線を維持 |

- エンジン管理ダイアログ（新規ダイアログ）の作成
- モデルの追加（ファイルインポート）機能

- カード型の一覧表示
- 星評価・おすすめ表示
- GPU設定ダイアログの分割（デバイス/エンジン分離）

### Phase 2: 将来の完全版で追加するもの

| 項目 | 内容 | 前提条件 |
|------|------|---------|
| エンジン管理ダイアログ | 左右分割、カテゴリ別セクション、カード風エントリ、右ペイン詳細 | モデル数が3個を超えるとき |
| モデル追加（インポート） | ファイルピッカーで `.onnx` を指定して追加 | インポート時の検証ロジック設計 |
| 自作/追加/標準の種別表示 | セクション分け + バッジ | 種別が複数存在するとき |
| おすすめ表示・星評価 | 標準エンジンの「おすすめ」バッジ、速度/精度の★評価 | エンジンのメタ情報定義 |

| 学習対応バッジ | カードに「学習対応」表示 | P4-1/P4-2 設計後 |
| ユーザーメモ欄 | 各エンジンに自由記述メモ | 外部モデル追加機能後 |
| カテゴリフィルタの永続化 | エンジンごとのカテゴリ選択を保存 | スコープ判断後 |

## 11. 実装前提メモ（AIエージェント向け）

### 固定すべき概念（変えてはならない）

1. **UI呼称は「検出エンジン」**: "model" "backend" "detector" は内部用語。ユーザー向けUIでは「検出エンジン」で統一する
2. **3種別の区分**: 標準 / 追加 / 自作。内部的には `origin: "bundled" | "imported" | "trained"` のような属性で管理
3. **日常切替と管理の分離**: コンボボックス（軽量・即座）と管理ダイアログ（詳細・比較）は別UI。統合しない
4. **教師データ管理4原則との整合**: `02_future-retraining-memo.md` の4原則を UI で表現する準備をする
5. **アダルト用語の非露出**: ラベルカテゴリの表示名に `genital_m` 等の内部名を出さない

### 後で変えてもよい部分

1. コンボボックスの配置（ツールバー or ステータスバー or メニュー）
2. 管理ダイアログの左右分割比率
3. 星評価のデザイン（★ or プログレスバー or テキスト）
4. セクションヘッダーのデザイン（罫線 or 背景色 or 余白）
5. カテゴリフィルタの位置（管理ダイアログ内 or 検出実行時のインライン表示）
6. 「おすすめ」の表現方法

### 実装前に追加調査が必要な点

1. **外部モデルのインポート仕様**: `.onnx` ファイルだけで動くのか、メタデータファイルが必要か。`DetectorFactory` の拡張設計
2. **エンジンメタデータのスキーマ**: 名前・説明・速度/精度・対応カテゴリ等をどこにどう持つか（JSON マニフェスト？ Python クラス属性？）
3. **カテゴリの汎用化**: `label_schema.py` のハードコードからモデルごとのカテゴリ定義への移行設計
4. **コンボボックスと GPU 設定ダイアログの連携**: 変更時の `GpuConfig` 保存タイミング・同期方法
5. **ツールバーの空きスペース**: UI リデザインで 15→10 ボタンに削減済み。コンボボックスの配置余地

### 今回あえて決めない方がよい点

1. **学習機能のUI詳細**: 管理ダイアログに「学習」セクションが入ることは想定するが、具体設計は P4-1/P4-2 時
2. **モデルマーケットプレイス**: 「モデルをダウンロードして追加」する導線は今は設計しない
3. **カテゴリフィルタの永続化スコープ**: セッション限り / プロジェクト単位 / グローバルのどれか
4. **バックエンド名と表示名のマッピング方式**: ハードコード / JSON設定 / アダプタ属性
5. **外部モデルの品質検証UI**: テスト検出ボタン等はインポート機能の設計と合わせて決める
