## 1. 今回決まったこと一覧

### 配布方針

| # | 決定事項 | 根拠 |
|---|---------|------|
| 1 | **検出モデルは原則として配布物に同梱しない** | モデルごとにライセンス条件が異なり、再配布の可否判断が重い。汎用ツールとして配布するうえで、特定モデルへの依存を避ける |
| 2 | **ユーザーが自分でモデルを取得し、追加する方式** | ライセンス責任をユーザー側に置く。アプリ側はモデルの「使い方」を提供し、「入手」は支援に留める |

## 2. App内から案内する公式配布ページの候補一覧

### 前提: App内リンクとは何か

App 内から案内できるのは「公式配布ページをブラウザで開く」操作のみ。以下は一切行わない。

- **直ダウンロードしない** — HTTP リクエストでモデルファイルを取得する機能は実装しない
- **ワンクリック取得しない** — ボタン一発でモデルが入る仕組みは作らない
- **ミラー配布しない** — 他所から取得したモデルをアプリが再配布しない
- **ライセンス保証しない** — リンク先のライセンスを保証・推奨する意図はない。ユーザーが自己責任で確認する

### 選定基準

- ライセンスが permissive（Apache 2.0, MIT, BSD 等）で商用利用が明示的に許可されている
- 一般用途（顔・物体・テキスト・ナンバープレート等）として説明可能
- 公式の配布ページが存在し、安定している
- ONNX 形式（または変換手順が整っている）
- UI 上で「おすすめ」「推奨」という強い表現を用いない

- 既存の `DetectorFactory` には `nudenet`, `nudenet_640m`, `erax_v1_1` が登録されている

- App 内からリンクしないだけであり、機能として削除するわけではない

## 4. SAM2 の位置づけ整理

### 基本方針

| 項目 | 内容 |
|------|------|
| **役割** | 輪郭エンジン（contour engine）。検出エンジンではない |
| **機能** | GrabCut の上位代替として、マスク輪郭の精緻化・品質向上を担う |
| **配布** | アプリに同梱する（セットアップ時にダウンロード） |
| **UI 露出** | メイン UI では前面に出さない。`contour_mode: quality` の内部実装 |
| **表記** | Version / About ウィンドウの Third-Party Components セクションに集約する。ただし「Version 情報への記載だけでよい」とは断定しない。Apache 2.0 は LICENSE 本文・NOTICE ファイルの配布物への同梱を求める可能性があるため、配布形態が確定してから最終判断する |

### 検出エンジンとの違い

```
検出エンジン（Detection Engine）
  → 映像フレームから対象領域を「発見」する
  → NudeNet, EraX, MediaPipe 等
  → ユーザーが選択・切替する
  → 原則同梱しない

輪郭エンジン（Contour Engine）
  → 検出された領域の「輪郭を精緻化」する
  → SAM2, GrabCut 等
  → ユーザーは直接選択しない（品質モードの内部実装）
  → SAM2 は同梱する
```

### 現在の実装状態

- `app/infra/ai/sam2_tiny_adapter.py`: SAM2 tiny ONNX アダプタ実装済み
- `app/domain/services/contour_service.py`: ContourService が SAM2 を品質モードで利用
- `scripts/install_deps.py`: セットアップ時に HuggingFace から ONNX モデルをダウンロード
  - `sam2_tiny_encoder.onnx` — SharpAI/sam2-hiera-tiny-onnx
  - `sam2_tiny_decoder.onnx` — 同上
- SAM2 不在時は GrabCut にフォールバック（既存動作）

### ライセンス確認事項（Apache 2.0）

SAM2 は Apache License 2.0 で公開されている。Apache 2.0 の主な義務:

| 義務 | 内容 | 対応状況 |
|------|------|---------|
| 著作権表示の保持 | 原著作権表示を配布物に含める | **要対応**: Version 情報に記載する（セクション5参照） |
| LICENSE テキストの同梱 | Apache 2.0 のライセンス全文を配布物に含める | **要確認**: 配布形態（インストーラ / zip / pip）に応じた同梱方法を決定する |
| NOTICE ファイルの同梱 | 元リポジトリに NOTICE がある場合、その内容を配布物に含める | **要確認**: SAM2 リポジトリの NOTICE ファイルの有無を確認する |
| 変更の明示 | 改変した場合はその旨を明示する | 該当なし（ONNX モデルをそのまま使用） |

> **確認タスク**: SAM2 の元リポジトリ（facebook/sam2）の NOTICE ファイルの有無と内容、および SharpAI/sam2-hiera-tiny-onnx の ONNX 変換版のライセンス継承状態を確認すること。配布物側の notices 同梱要否は、5月レビュー前に結論を出す。

### 各エントリの記載項目

| 項目 | 必須 | 内容 |
|------|:---:|------|
| コンポーネント名 | Yes | 正式名称 |
| 著作権表示 | Yes | Copyright (c) [年] [権利者] |
| ライセンス名 | Yes | Apache 2.0, MIT, LGPL v3 等 |
| 参照URL | Yes | 公式リポジトリまたはプロジェクトページ |
| 補足 | No | 用途の簡潔な説明（必要な場合のみ） |

### 設計原則

1. **技術寄りすぎない**: 「ONNX Runtime CUDAExecutionProvider」ではなく「ONNX Runtime」
2. **必要情報は落とさない**: 著作権表示・ライセンス名・URLは必ず記載
3. **拡張可能**: 将来同梱物が増えた場合、同じフォーマットで追加するだけ
4. **ライセンス全文へのアクセス**: 「ライセンス全文を表示」ボタンで、各ライセンスのフルテキストを表示可能にする（テキストファイルを開く or ダイアログ内表示）
5. **同梱コンポーネントのみを載せる**: ユーザーが追加した検出モデルは Third-Party Components に**自動列挙しない**。理由: ユーザーが追加したモデルのライセンス・著作権表示を App が代わりに掲載することは、App がそのモデルを「保証している」という誤解を生む。検出モデルのライセンス表示はユーザー自身の責任範囲とする

### 将来の拡張

同梱コンポーネントが増えた場合:

```
■ [コンポーネント名]
  Copyright (c) [年] [権利者]
  Licensed under [ライセンス名]
  [URL]
```

このフォーマットを繰り返すだけで済む。カテゴリ分け（「推論エンジン」「UIフレームワーク」等）は、コンポーネント数が 10 を超えた段階で検討する。現時点では不要。

## 6. モデル管理導線の方針（将来の管理 UI 要件）

以下は Phase 2 以降で設計・実装するエンジン管理ダイアログに必要な導線の整理。5月レビュー時点では実装しない。

### 導線一覧

| # | 導線 | 説明 | Phase |
|---|------|------|:-----:|
| 1 | **公式配布ページを開く** | 一般用途モデルの公式配布ページ一覧を表示 → クリックでブラウザが開く。アプリ内ダウンロードは行わない | 2 |
| 2 | **ファイルから追加** | ファイルピッカーで `.onnx` を選択 → `models/` にコピー → レジストリに登録 | 2 |
| 3 | **モデル保存先を開く** | `models/` ディレクトリをファイルマネージャーで開く。手動配置の補助 | 2 |
| 4 | **再スキャン / リロード** | `models/` を再スキャンし、未登録の `.onnx` を検出してリストに反映 | 2 |
| 5 | **手動配置モデルの再認識** | 手動で `models/` に配置したファイルをアプリに認識させる（再スキャンの一形態） | 2 |
| 6 | **モデルのエクスポート** | モデル定義（メタデータ + ファイルパス参照）をエクスポート。共有・バックアップ用 | 将来 |
| 7 | **モデルのインポート** | エクスポートしたモデル定義を取り込む | 将来 |

### 導線のフロー（Phase 2 設計案）

```
エンジン管理ダイアログ
  │
  ├── [+ エンジンを追加...]
  │     │
  │     ├── 「一般用途モデルの公式配布ページ一覧を見る」
  │     │     → 公式ページ候補を表示（ブラウザで開くのみ）
  │     │     → ユーザーが自分でダウンロードして models/ に配置
  │     │
  │     ├── 「ファイルから追加」
  │     │     → ファイルピッカー (.onnx)
  │     │     → models/ にコピー
  │     │     → メタデータ入力（名前・説明・対象カテゴリ）
  │     │
  │     └── 「モデルフォルダを開く」
  │           → models/ をエクスプローラで開く
  │           → 手動配置後「再スキャン」
  │
  └── [再スキャン]
        → models/ を走査
        → 未登録の .onnx を検出
        → 登録ウィザード or 自動登録
```

### 重要な制約（固定）

- **アプリ内から直接ダウンロードしない**: HTTP リクエストでモデルファイルを取得する機能は実装しない
- **直接ファイル URL を案内しない**: `.onnx` の直リンクは一覧に入れない。公式の配布ページのみ
- **ライセンスを保証しない**: リンク先のモデルを「互換確認済み」として保証・推奨する表現を使わない
- **メタデータは最低限**: 初回追加時にユーザーが入力するのは「名前」と「説明」のみ。ラベル定義は自動検出を試み、失敗時は手動設定

### エクスポート / インポートの意味境界（Phase 2 設計前の確認事項）

エクスポート / インポートは「モデルの持ち運び」機能として将来設計するが、モデルの種別によって扱いが異なる。実装前にこの境界を明確にしておく。

#### 種別ごとの持ち運び可能性

| 種別 | エクスポート対象 | 注意点 |
|------|:----------:|------|
| **自作モデル**（ローカル学習済み） | **対象になりうる** | ファイル本体 + メタデータをまとめてエクスポートできる設計を目指す。ただし学習元データの権利・ファイルサイズ等の実務的制約がある |
| **外部追加モデル**（ユーザーが自分で持ち込んだ `.onnx`） | **原則として対象外または警告付き** | 後述 |
| **標準同梱モデル**（SAM2 等） | **対象外** | 配布物に含まれる。エクスポートの概念が当たらない |

#### 外部追加モデルの export が難しい理由

外部から持ち込んだモデルは、以下のいずれかの問題を持つことがある。

1. **ライセンスがファイル配布を禁じている**: Apache 2.0 でも NOTICE の扱いが不明確な場合がある。非 permissive ライセンスなら再配布禁止の可能性がある
2. **ファイル本体の由来が不明**: ユーザーが自分で変換した ONNX や、非公式ミラーから取得したファイルは来歴が不明
3. **App はライセンスを検証できない**: ファイルの内容からライセンスを機械的に判定する手段がない

そのため、外部追加モデルの export については将来のいずれかの設計を選択することになる。

| 設計オプション | 説明 | 推奨度 |
|-------------|------|:-----:|
| **非対応** | 外部追加モデルはエクスポート対象にしない | 安全だが利便性が低い |
| **警告付きエクスポート** | 「このモデルのライセンスをご自身で確認してください」と表示した上でファイルをエクスポート | バランス的 |
| **参照情報のみ** | ファイル本体はエクスポートせず、メタデータ（名前・説明・元ファイルパス参照）のみをエクスポート | 安全。再インポート時にファイルが別途必要 |

> **方針**: 外部追加モデルの export 設計は Phase 2 設計時に確定する。現時点では上記3オプションのいずれかになることを前提に、モデルメタデータとファイルパスを分離して管理できる構造を確保しておくこと。

#### App が保証しないこと

以下は App として保証・担保しない。設計・実装時に混入させないこと。

- 外部追加モデルのライセンスの適法性
- エクスポートしたファイルを第三者へ渡すことの適法性
- インポートしたモデルが元の公式配布版と同一であること

## 7. PT → ONNX 変換経路の確認結果

### 現状の実装

| 項目 | 状態 |
|------|------|
| **変換機能の有無** | **あり** — `scripts/install_deps.py` 内の `_export_erax_to_onnx()` |
| **対象モデル** | EraX (YOLO11 形式) のみ |
| **変換方法** | `ultralytics.YOLO(...).export(format="onnx", imgsz=640, opset=12, dynamic=False)` |
| **変換タイミング** | セットアップ時（`setup.bat` → `install_deps.py`）のみ |
| **UI からの到達** | **可** — 「検出モデル管理」ダイアログで EraX が「変換が必要」状態のとき変換ボタンを表示（2026-04-14 追加） |
| **現在の動作可否** | **動作する** — ただし `ultralytics` パッケージのインストールが前提 |
| **汎用性** | **低い** — EraX 専用のパラメータ（imgsz=640, opset=12）がハードコード |

### 詳細

変換ロジックは `app/infra/ai/erax_export.py` に EraX 専用サービスとして切り出し、
`scripts/install_deps.py` と「検出モデル管理」ダイアログの両方から再利用する。

```python
# app/infra/ai/erax_export.py
def export_erax_pt_to_onnx(pt_path, onnx_path, *, imgsz=640, opset=12) -> Path:
    from ultralytics import YOLO
    model = YOLO(str(pt_path))
    exported = model.export(format="onnx", imgsz=imgsz, opset=opset, dynamic=False)
    # … 出力を onnx_path へリネーム …
    return onnx_path
```

- `ultralytics` 未インストール時は変換できず、UI ではボタンが無効化される
- **`EraXAdapter` は `.onnx` のみ読み込む**。`.pt` をランタイムで直接読む fallback は実装していない
- `erax_v1_1.pt` のみ存在し `.onnx` が無い場合、EraX 検出は利用できない（管理 UI 上は「変換が必要」と表示）
- この変換は EraX v1.1 専用であり、任意 `.pt` の汎用変換は提供しない

### 将来のモデル追加フローへの再利用性

| 観点 | 評価 |
|------|------|
| YOLO 系モデルの変換 | **再利用可能** — ultralytics の export 機能は YOLO 系全般に使える |
| 非 YOLO 系モデル | **再利用不可** — `torch.onnx.export()` による汎用変換は未実装 |
| ユーザー向け UI | **未実装** — 変換はセットアップスクリプト限定 |
| 将来の学習モデル | **計画あり** — `03_erax-local-retraining-implementation-plan.md` で ONNX export を想定 |

### 5月レビュー時点での扱い

- PT → ONNX 変換機能を **レビュー時点で前面に出す必要はない**
- 現状の EraX 用変換は内部機能として存在し、動作する
- 汎用的な「モデル変換ツール」の UI 化は Phase 2 以降
- **確認結果**: 変換経路は「EraX 専用で生きている」が「汎用ではない」状態

## 9. AIエージェント向け実装前提メモ

### 固定すべき決定事項（変えてはならない）

1. **検出モデルは配布物に同梱しない**（SAM2 を除く）
2. **App内リンクは公式配布ページをブラウザで開くのみ**。直ダウンロード・ワンクリック取得・ミラー配布は実装しない。ライセンス保証も行わない
3. **App内から案内する公式配布ページは permissive ライセンスの一般用途モデルのみ**
4. **SAM2 は輪郭エンジンとして同梱**。検出エンジン一覧には含めない
5. **UI 呼称は「検出エンジン」**（`05_model-switcher-ui-design.md` セクション2 準拠）
6. **3種別の区分: 標準 / 追加 / 自作**（内部: `bundled / imported / trained`）
7. **教師データ管理4原則**（`02_future-retraining-memo.md` 準拠）
8. **アダルト用途のラベル・モデル名は UI に露出しない**
