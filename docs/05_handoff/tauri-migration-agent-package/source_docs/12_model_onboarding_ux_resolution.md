## 1. この文書で解くこと

現在の文書群には、以下の 2 つの設計意図が混在している。

1. **レビュー到達のため、検出モデルを簡単に使える状態にしたい**
2. **製品としては、検出モデルの同梱・取得・保証にアプリが踏み込みすぎない**

この混在により、UI 導線が次のように揺れている。

- 「検出モデルは非同梱」としながら、`setup.bat` では自動取得を試みる
- 「App 内で直ダウンロードしない」としながら、「エンジンを追加」導線が未定義
- モデル種別の UI 区分がある一方で、ユーザーは実際には自分が入手したモデル名で認識する
- 「おすすめ」「公式ページ案内」が、アプリによる推奨・保証と誤解される可能性がある
- SAM2 は同梱するが、検出エンジンではないため、モデル管理 UI に入れると混乱する

本書では、これらを **製品版の UX 方針** と **5月レビュー向けの暫定運用** に分ける。

## 4. 製品版 UX の固定方針

### 4.1 初回状態

検出モデルがない状態でも、アプリは通常起動できる。

ユーザーに見せるべき状態:
- 動画を開く
- 手動マスクを追加する
- 既存プロジェクトを編集する
- 書き出す
- 検出エンジンが未準備のため、自動検出はまだ使えない

避けるべき状態:
- 起動直後にエラー扱いする
- 「モデルをダウンロードしてください」と命令する
- 特定モデル名や内部ファイル名を前面に出す
- 検出できない理由を ONNX / session / backend など技術語だけで説明する

推奨文言の方向:

```text
自動検出を使うには、検出エンジンのモデルファイルを追加してください。
モデル未追加でも、手動マスク編集と書き出しは利用できます。
```

### 4.2 Phase 1（レビュー時点）の導線

5月レビュー時点では、完全な管理ダイアログを作らない。

残してよい導線:
- ツールバーの検出エンジン選択
- ステータスバーの使用中エンジン表示
- 対象カテゴリの最小フィルタ
- 詳細設定へのリンク
- モデル未導入時の説明

入れない導線:
- App 内のモデルダウンロード
- モデル追加ウィザード
- 公式配布ページ一覧
- モデル種別の本格的なカード UI
- 学習対応の詳細 UI

### 4.3 Phase 2（レビュー後）の導線

Phase 2 では、エンジン管理ダイアログを設計する。

必要な入口:
- `ファイルから追加`
- `モデルフォルダを開く`
- `再スキャン`
- `公式配布ページを見る`

禁止する入口:
- `今すぐダウンロード`
- `.onnx` 直リンクを押させる導線
- `おすすめモデルを取得`
- ライセンス確認をアプリが代行するような導線

管理ダイアログの基本構造:

```text
検出エンジン管理
  左: 登録済みモデル一覧
    - モデル名を主表示
    - 追加 / 自作 / 既知アダプタ等の出所は補助表示
  右: 選択中エンジンの詳細
    - 表示名
    - 用途説明
    - モデルファイル状態
    - 対象カテゴリ
    - 学習対応（属性）
    - 由来メモ
  下部:
    - ファイルから追加
    - モデルフォルダを開く
    - 再スキャン
    - 公式配布ページを見る
```

## 5. 用語ルール

| 内部語・避けたい語 | UI で使う語 | 備考 |
|------------------|------------|------|
| detector backend | 検出エンジン | ユーザー向け標準語 |
| model | モデルファイル / 検出エンジン | 文脈に応じて使い分ける |
| bundled | 原則 UI 主表示にしない | 検出モデルでは「標準」「同梱」を主表示にしない。必要なら内部属性としてだけ扱う |
| imported | 追加モデル | ユーザーが入れたもの。主表示はモデル名 |
| trained | 自作モデル | ローカル学習済み。主表示はユーザーが付けたモデル名 |
| label | 対象カテゴリ | ML 用語を避ける |
| confidence threshold | 検出の厳しさ | 詳細設定内のみ |
| ONNX / CUDA / backend | 詳細情報 | 日常導線には出さない |
| recommend / おすすめ / 推奨 | 候補 / 公式ページ / 参考 | 外部モデル案内では禁止寄り |
| 内部ID / 用途依存ラベル | モデル名 / ユーザーが付けた表示名 | モデル名は主表示にしてよい。ただし内部IDやアダルト用途ラベルを製品の第一印象として前面化しない |

## 6. ラベル辞書・対象カテゴリ方針

モデルをユーザーが導入する方式にすると、モデルごとに利用できるラベルが変わる。そのため、対象カテゴリは固定 UI ではなく、以下の3層で扱う。

```text
モデル固有の生ラベル
  例: モデルAの raw label
  ↓
アプリ内の正規化カテゴリ
  例: face / plate / text / person など
  ↓
ユーザー向け表示名
  例: 顔 / ナンバープレート / 文字 / 人物
```

採用方針:
- アプリ内に **共通ラベル辞書** を持つ
- 既知モデルごとに **raw label → 正規化カテゴリ** の対応表を持つ
- UI は正規化カテゴリをユーザー向け表示名へ変換して出す
- モデル切替時は、そのモデルが対応しているカテゴリだけを表示する
- 古い設定に現在モデル非対応のカテゴリが残っていても、保存値を壊さず、適用時に無視する
- unknown / imported custom model では、ラベル対応が分からない限りカテゴリフィルタを自動で有効化しない
- 未知モデルは、将来の管理 UI で「ラベル対応を設定する」導線を設ける

避けること:
- すべてのモデルに同じ固定ラベル UI を出す
- モデルに存在しないカテゴリを有効化できる状態にする
- モデルの raw label をそのまま製品 UI の第一印象として出す
- アプリが未知モデルの意味を推測で断定する

Claude への設計指示:
- Phase 1 では既知モデルの対応表だけでよい
- Phase 2 ではモデル追加時にラベル対応を確認・編集できる設計を検討する
- ラベル辞書は「モデル名表示」と独立させる。モデル名が同じでも、ラベル対応はメタデータとして扱う

## 7. レビュー向け暫定運用の扱い

`04_review/02_model-setup-guide.md` にある `setup.bat` の自動取得は、レビューを成立させるための暫定運用として扱う。

Claude は、この暫定運用を **製品版 UI の恒久設計に持ち込まない** こと。

レビュー資料では許容すること:
- `setup.bat` がモデル取得を試みると説明する
- 手動配置先として `models/` フォルダを示す
- ファイル名を必要最小限で示す
- モデル未導入時にできること / できないことを説明する

レビュー資料でも避けたいこと:
- 特定外部モデルを強く推奨する
- App がライセンスを確認済みであるように書く
- 製品版でも自動取得が標準導線であるように書く
- 製品の第一印象がアダルト専用になる説明に寄せる

## 9. 受け入れ基準

Claude の設計結果は、以下を満たす必要がある。

- 検出モデル非同梱と、レビュー向け暫定自動取得が明確に分離されている
- モデル未導入状態がエラーではなく、編集可能な空状態として設計されている
- App 内で直ダウンロード・ワンクリック取得・ミラー配布が発生しない
- 外部モデルを推奨・保証する表現がない
- 検出モデルに対して「標準」「同梱」を主表示していない
- SAM2 が検出エンジンではなく輪郭エンジンとして扱われている
- モデル名主表示を基本にし、追加 / 自作 / 学習対応の区分がユーザーに過剰な専門知識を要求しない
- モデルごとに利用できる対象カテゴリの違いが、ラベル辞書・対応表・UI表示で破綻しない
- 既知モデルは raw label → 正規化カテゴリ → ユーザー向け表示名の対応が定義されている
- unknown / imported custom model でラベル意味を推測断定していない
- 製品 UI がアダルト専用に見えない
- Phase 1 と Phase 2 の境界が明確で、5月レビュー前に過剰実装を求めない
- 既存の `05_model-switcher-ui-design.md` 案Bを活かしつつ、新しい配布方針に合わせて修正されている

## 11. PM 判断

現時点で Claude に設計させるべき範囲は、**モデル導入・管理 UX の情報設計とワイヤーフレーム** までである。

今すぐ実装仕様として確定させるべきではない。理由は以下。

- 配布・ライセンス方針の制約が強く、文言ひとつで製品の見え方が変わる
- 5月レビュー向け暫定導線と製品版導線を混同すると、後で設計を戻すコストが大きい
- Phase 1 の目的は「レビュー者にモデル差し替え前提が伝わること」であり、完全な管理 UI ではない
- Phase 2 の管理 UI はレビュー後のフィードバックで変わる可能性が高い

したがって、Claude には **実装指示ではなく、制約に沿った UX 方針・状態設計・文言案・ワイヤーフレーム** を出させる。
