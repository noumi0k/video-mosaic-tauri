# Auto Mosaic Tauri 完全新規実装 指示書（AIエージェント向け）

> 位置づけ: このファイルは from-scratch 方針の設計履歴です。現在の作業正本ではありません。現行実装の責務境界と不変条件は `docs/engineering/current-implementation.md` を優先してください。

## 0. この文書の目的

この文書は、**AIエージェントが Auto Mosaic を Tauri + Python 構成で完全新規実装するための実装指示書**である。  
既存の PySide6 実装や途中移行コードは、**実装の土台として再利用しない**。  
必要に応じて要件・思想・対象機能の確認資料として参照してよいが、**コードや依存構成、起動導線、セットアップ導線、UI設計の直接流用は禁止**とする。

この方針を採る理由は明確である。

- 既存コード参照のまま移行を進めた結果、導入段階でエラーが頻発した
- CUDA / ONNX Runtime / Python 実行環境 / UI 境界の整理が不十分なまま複雑化した
- PySide6 前提の UI 思考や過去レイアウトを引きずり、Tauri 向けの自然な設計になりにくかった
- AIエージェントによる継続実装では、「何を捨てて何を残すか」が曖昧だと失敗しやすい

したがって本プロジェクトは、**参考資料は読むが、実装は完全にゼロから設計・構築する**。

## 1. 最重要原則

### 1-1. 実装は完全新規

以下は禁止する。

- 既存 PySide6 コードのコピー＆ペースト
- 既存ディレクトリ構成の踏襲
- 既存 setup.bat / run_ui.bat / runtauri.bat の改造継続
- 既存 MainWindow や UI ロジックの移植
- 既存の「とりあえず動いているコード」を前提にした依存の温存

以下は許可する。

- 製品要求の確認
- マスクトラック要求の確認
- 学習モデル候補の確認
- 将来機能メモの確認
- 過去に起きた失敗の再発防止のための参照

要するに、**既存コードは設計資料であって、実装資産ではない**。
## 実装優先方針（AIエージェント向け上書き）

本案件では GPU / CUDA 高速化を後回しにせず、初期段階から正式要件として扱う。
ただし、GPU 初期化失敗でアプリ全体が起動不能になってはならない。

そのため、以下を必須とする。

- GPU 診断機能を UI 起動や本処理と分離する
- Python CLI として `doctor`, `gpu-status`, `video-probe` を先に実装する
- Tauri とは subprocess + JSON I/O で接続する
- CPU ベースラインは保持するが、設計の主眼は GPU 高速化対応に置く
- CUDA / onnxruntime-gpu / torch / Python の整合性を先に定義する

### 1-2. まず「壊れにくい起動と実行基盤」を作る

本プロジェクトでは、検出精度や高機能UIより先に以下を成立させること。

1. 環境診断ができる
2. 必要ランタイムが足りない場合、理由を明示できる
3. CPU モードで最低限動く
4. GPU / CUDA が使える場合のみ加速する
5. UI はバックエンド未完成でも壊れず起動する
6. 検出・書き出しなどの長時間処理は job として分離される

### 1-3. CUDA は必須要件ではなく、加速オプション

CUDA / GPU は重要だが、**起動不可の原因にしてはならない**。

必須条件:

- GPU がなくても CPU で起動可能
- CUDA が壊れていても CPU で起動可能
- ONNX Runtime GPU が使えない場合、自動または明示的に CPU にフォールバック
- UI が GPU 初期化失敗で巻き込まれて落ちない

### 1-4. UI は Tauri に最適化し、過去UIを見ない

過去の PySide6 UI は参照禁止ではないが、**レイアウト判断の根拠にしない**こと。

UI 設計は以下を前提に新しく行う。

- Tauri + Web UI として自然であること
- 状態管理がしやすいこと
- 動画編集ソフト風の構造を持ちながら、実装が破綻しないこと
- 画面都合で backend 設計を歪めないこと

## 2. プロダクトの目的

このソフトの目的は以下である。

- ローカル動画を読み込む
- AI により対象箇所の初期マスク候補を生成する
- 候補を時間軸上の **mask track** として保持する
- ユーザーがプレビュー上・タイムライン上でマスクを修正する
- 修正結果を保ったままモザイク付き動画を書き出す
- プロジェクトを保存し、後で再編集できる

設計思想は以下。

- 完全ローカル実行
- オフライン前提
- AI は候補生成の補助であり、最終品質はユーザーが担保する
- 漏れ防止を優先し、必要なら広めのマスクを許容する
- 編集単位は断片マスクではなく **mask track** である

## 4. 今回の非交渉ルール

### 4-1. 既存実装を前提にしない

AIエージェントは、過去コードに「足りない部分を継ぎ足す」発想を捨てること。

常に以下の問いで進めること。

- この機能をゼロから作るなら、最も単純で保守しやすい構造は何か
- Tauri 時代に不要な PySide6 的発想を持ち込んでいないか
- 既存の失敗要因を再導入していないか

### 4-2. セットアップスクリプトに複雑なことをさせない

禁止:

- setup スクリプト内で過剰な分岐を行う
- バッチファイルに大量のロジックを書く
- setup 時に暗黙で複数の重要処理を抱え込む

推奨:

- setup は開発用 / 本番用で責務分離
- 複雑な処理は Python 側の明示的コマンドに寄せる
- バックエンド側に `doctor` / `check-env` / `install-models` / `benchmark` のような CLI を用意する

### 4-3. UI と処理を混ぜない

- UI は状態表示と操作入力に集中する
- 検出・補間・書き出し・学習・教師データ保存は Python backend が担当する
- Tauri 側は「実行要求」「進捗受信」「結果反映」を担う

### 4-4. 長時間処理はすべてジョブ化

以下は同期実行禁止。

- 動画全体の自動検出
- 書き出し
- 教師データエクスポート
- 自動再学習
- モデル変換

これらはすべて job として扱い、状態を持たせる。

## 6. フロントエンドと Python のつなぎ方（最重要）

ここは特に詳細に実装すること。

## 6-1. 基本方針

**最初の正式方式は subprocess + CLI + JSON I/O** とする。

ローカル HTTP サーバーを最初から建てない。  
WebSocket 常駐サーバーも最初から前提にしない。  
まずは、**壊れにくくデバッグしやすい最小構成**を採る。

## 6-2. 通信方式の原則

Tauri 側は Python を以下のどちらかで呼ぶ。

1. 単発コマンド実行
2. 長時間ジョブ実行

### 単発コマンドの例
- 環境チェック
- 動画メタ情報取得
- 設定ファイル読込
- モデル一覧取得
- プロジェクト読込

### 長時間ジョブの例
- 全体検出
- 書き出し
- 自動再学習
- 教師データ生成

## 6-3. JSON 契約

Python CLI は、**入力 JSON を受け取り、出力 JSON を返す**こと。

例:

```json
{
  "command": "video.get_metadata",
  "args": {
    "video_path": "C:/videos/sample.mp4"
  }
}
```

応答例:

```json
{
  "ok": true,
  "data": {
    "width": 1920,
    "height": 1080,
    "fps": 29.97,
    "frame_count": 12345,
    "duration_sec": 412.0
  },
  "error": null
}
```

失敗例:

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "VIDEO_OPEN_FAILED",
    "message": "Failed to open video.",
    "details": {
      "path": "C:/videos/sample.mp4"
    }
  }
}
```

## 6-4. 標準出力 / 標準エラーの扱い

- **stdout は JSON 応答専用**
- ログや進捗は stderr またはログファイルへ出す
- フロントエンドは stdout の JSON のみを正式な応答として扱う
- Python 例外の traceback をそのままユーザー表示しない

## 6-5. ジョブ実行方式

長時間処理は以下の流れで統一する。

1. Tauri → Python にジョブ開始要求
2. Python は `job_id` を返す
3. Python 側でジョブ状態を保存する
4. Tauri は定期的に `job.get_status` を呼ぶ
5. 完了時に成果物パスや結果 JSON を返す

ジョブ状態例:

- queued
- running
- succeeded
- failed
- cancelled

## 6-6. ジョブステータス JSON 例

```json
{
  "ok": true,
  "data": {
    "job_id": "job_20260402_0001",
    "status": "running",
    "progress": 42.5,
    "stage": "detection",
    "message": "Processing frame 850 / 2000"
  },
  "error": null
}
```

## 6-7. キャンセル設計

- 書き出し
- 自動検出
- 自動再学習

これらはキャンセル可能にすること。

Python 側はジョブごとにキャンセルフラグを持ち、ループ中で定期確認する。

## 6-8. 共有データ形式

フロントエンドとバックエンドでやり取りする主要構造は、**TypeScript 型と Python モデルを対応表で管理**すること。

最低限、以下は共通契約化する。

- Project
- VideoMeta
- MaskTrack
- Keyframe
- MaskStyle
- JobStatus
- DetectorConfig
- ExportConfig
- DatasetCaptureConfig
- RetrainConfig

## 6-9. パスの扱い

- フロントエンドは OS パスを好き勝手に結合しない
- パス解決は Python backend が責任を持つ
- Tauri 側は選択されたパスを渡すだけに近づける

## 6-10. なぜこの方式を採るのか

理由:

- デバッグしやすい
- 途中で Python 単体診断ができる
- UI が壊れても backend 単体テスト可能
- 後から HTTP / IPC 常駐化する余地がある
- AIエージェントにとって責務分離が明確で失敗しにくい

## 8. 機能仕様

## 8-1. 動画読み込み

必須要件:

- MP4 / MOV / WebM を優先対応
- メタ情報取得に失敗した場合、理由を表示
- 動画が読めない場合もアプリ全体は落ちない
- フロントエンドは動画メタ情報とサムネイルを表示可能

## 8-2. 自動検出

標準方式:

- NudeNet 320n をデフォルト
- 推論解像度は設定可能
- 検出閾値を設定可能
- フレーム間引き設定を可能にする
- GPU 利用の可否を設定・自動判定できるようにする

任意方式:

- EraX backend
- NudeNet 640m

要件:

- モデルが無い場合はエラーではなく未利用状態として扱う
- 検出器切替で後段データ形式が変わらないこと

## 8-3. マスクトラック

編集単位は **mask track** とする。

必須要件:

- 1対象につき、できるだけ1本のトラックを維持
- 一時的に検出が途切れても即消えない
- manual keyframe は自動処理で上書きしない
- UI / 内部データ / 書き出しが同じ存在期間を共有する

内部状態:

- active
- lost
- inactive

キーフレーム source:

- auto
- manual
- interpolated
- predicted
- re-detected

## 8-4. 編集 UI

必須:

- トラック一覧
- プレビュー表示
- タイムライン
- キーフレーム表示
- 位置移動
- サイズ変更
- 多角形編集
- 表示 / 非表示
- トラック削除
- 手動トラック追加

## 8-5. モザイク

- モザイク強度変更
- expand_px 変更
- feather は対応余地を残す
- プレビューと最終書き出しで意味が一致すること

## 8-6. 書き出し

- MP4 / MOV / WebM
- H.264 を主軸
- 元動画 / 720p / 1080p / 4K
- ビットレート設定
- GPU エンコードは optional
- キャンセル可能
- 書き出しキュー対応

## 8-7. プロジェクト保存

保存対象:

- source video path
- video meta
- mask tracks
- keyframes
- export config
- detector config
- dataset capture config
- retrain config

## 9. 教師データ保存の本実装

これは今回、将来メモではなく**正式機能として実装**する。

ただし、以下の原則を厳守すること。

### 9-1. 基本原則

- デフォルト OFF
- ユーザーの明示的な opt-in が必要
- 保存先を UI で確認できる
- いつでも削除できる
- 外部送信は行わない

### 9-2. 保存対象

最低限保存するもの:

- project_id
- video_id
- frame_index
- model_name
- model_version
- inference_config
- initial detection result
- final edited result
- diff metadata
- human_verified flag

### 9-3. 保存単位

候補:

- frame 単位
- crop 単位
- mask 単位
- project 単位の index

推奨:

- 動画全体ではなく **必要最小限の crop + annotation + metadata**

### 9-4. 削除導線

ユーザーは以下をできること。

- 今回のプロジェクト分だけ削除
- 特定日付の教師データを削除
- すべての教師データを削除

### 9-5. UI要件

- 「教師データ保存: OFF / ON」が一目で分かる
- 保存先ディレクトリを表示できる
- 現在の件数と容量を確認できる

## 10. 自動再学習の本実装

これも今回、正式機能として扱う。  
ただし、**外部送信なし・ローカル学習のみ**を前提とする。

### 10-1. 基本原則

- デフォルト OFF
- 学習開始は明示操作のみ
- バックグラウンド job として実行
- 途中キャンセル可能
- 失敗時に既存モデルを壊さない

### 10-2. 学習対象

第一段階では、完全な end-to-end 大規模再学習ではなく、以下の現実的方式を優先すること。

- 軽量ファインチューニング
- detector の補助 head 学習
- 小規模再学習
- 既存モデルとの差分モデル生成

### 10-3. 学習前チェック

- 学習データ件数
- GPU 利用可否
- VRAM 量
- 空きディスク容量
- 出力先
- 学習予定時間の概算

### 10-4. 学習ジョブ要件

表示すべき状態:

- queued
- running
- validating
- exporting
- succeeded
- failed
- cancelled

### 10-5. モデル管理

- 既存安定モデルを残す
- 新モデルはバージョン付きで保存
- 既定モデル切替は手動承認制
- 失敗した学習結果を既定モデルにしない

### 10-6. 最初の現実的ゴール

- 「編集結果を材料にローカル再学習できる」
- 「その結果を別モデルとして保存できる」
- 「ユーザーが使用モデルを切り替えられる」

外部共有や連合学習は今回やらない。

- blocked
- warning
- ready

blocked の例:

- Python backend 起動不可
- 必須パッケージ欠落
- 設定フォルダ書込不可

warning の例:

- GPU 利用不可だが CPU 利用可
- optional model 未配置
- ffmpeg が見つからないため書き出し制限あり

ready の例:

- CPU or GPU で必要機能が成立

---

## 12. インストーラ設計

インストーラは、ただファイルを置くだけではなく、**壊れにくい導入と削除**を保証すること。

### 12-1. 目標

- ユーザーが迷わず導入できる
- 起動後すぐに doctor が走る
- モデル配置状態が分かる
- どこに何が入ったかが追跡できる
- アンインストールが確実にできる

### 12-2. 配布構成の原則

分離対象:

- アプリ本体
- Python backend 実行環境
- モデル
- ユーザーデータ
- ログ
- 一時ファイル

### 12-3. 推奨方式

- アプリ本体とユーザーデータを分離
- モデルは共通配置または user data 配下で管理
- 設定・ログ・一時ファイルは OS 標準のアプリデータ領域を優先

### 12-4. インストーラが行うこと

- 本体配置
- 必要ショートカット作成
- アンインストーラ登録
- 初回起動用設定生成
- モデル未配置時の導線準備

### 12-5. インストーラが行わないこと

- 強引な GPU ドライバ導入
- CUDA 本体の再配布
- 破壊的な既存環境上書き

## 13. アンインストール設計（追加必須）

本仕様では、アンインストール方法を正式要件に含める。

## 13-1. 基本原則

- アンインストールは確実であること
- 本体削除だけでなく、残存データの扱いを選べること
- ユーザーが「何が残るか」を理解できること

## 13-2. アンインストール種別

### A. 標準アンインストール
削除対象:
- アプリ本体
- ショートカット
- 登録情報

保持対象:
- プロジェクトファイル
- ログ
- 設定
- モデル
- 教師データ
- 再学習済みモデル

### B. 拡張アンインストール
削除対象:
- 標準アンインストール対象
- 設定
- ログ
- temp
- cache

保持対象:
- プロジェクト
- 教師データ
- 再学習済みモデル

### C. 完全アンインストール
削除対象:
- 標準 + 拡張の全対象
- モデル
- 教師データ
- 再学習済みモデル
- job 履歴
- export キャッシュ

## 13-3. UI要件

アンインストーラ、または設定画面から以下を選べるようにする。

- 本体のみ削除
- 本体 + 設定/ログ削除
- 完全削除

## 13-4. OS導線

Windows では最低限以下に対応すること。

- スタートメニューからアンインストール
- 「インストール済みアプリ」からアンインストール
- 必要に応じてアンインストーラ実行ファイル直起動

## 13-5. 削除前確認

完全削除時には確認すること。

- 削除件数
- 削除対象ディレクトリ
- 教師データ削除有無
- 再学習済みモデル削除有無
- プロジェクトファイルは通常保持するか確認

## 13-6. ログ

アンインストール実行時は以下を残すか、ユーザーが選択できること。

- 最終アンインストールログ
- 削除失敗ファイル一覧

## 17. AIエージェントへの作業姿勢

AIエージェントは以下の姿勢で進めること。

1. まず起動基盤を作る
2. 次に CLI JSON 契約を固定する
3. 次に Project / Track / Keyframe を固める
4. その後に検出・編集・書き出しへ進む
5. CUDA 最適化は最後ではないが、起動基盤を壊してまで先にやらない
6. 常に CPU fallback を維持する
7. 問題が出たら「既存コードと違うから」ではなく「新規設計として責務が曖昧ではないか」で見直す

## 18. 最終目標

最終的に目指す状態は以下。

- Tauri フロントエンドが安定して起動する
- Python backend が独立して診断・実行できる
- GPU が使えれば速い
- GPU が壊れていても CPU で動く
- ユーザーは断片マスクではなく mask track を編集する
- 編集結果を保存・再読込できる
- モザイク動画を書き出せる
- 教師データをローカル保存できる
- ローカル自動再学習ができる
- 導入と削除が明確で、アンインストールも安全に行える

## 19. エージェントへの最終指示

このプロジェクトでは、**既存コードベースの移植や改修ではなく、完全新規実装として判断すること。**

判断基準は常に以下である。

- 今ゼロから作るならどう設計するか
- AIエージェントが段階実装しても破綻しないか
- 導入時にエラーが連鎖しにくいか
- Tauri と Python の責務が明確か
- CPU fallback を維持できているか
- 将来のモデル差し替え、教師データ保存、再学習、アンインストール設計まで破綻なく接続できるか

この文書を、以後の実装判断の基準とすること。
