## 1. システムの目的

Auto Mosaic は、動画の中に映り込んだ「隠したい領域」に対して、AI を補助として使いつつ、最終的には人間が責任を持って編集・確認したモザイクを書き出すための **ローカル動画モザイクエディタ** です。

設計の出発点となる価値観:

1. **AI は最終結果ではなく出発点を作る** — 自動検出は「ここが候補です」を提示するだけで、最終的な見え方の責任はユーザーにある。
2. **手で直したものは絶対に守る** — ユーザーが手作業で調整した内容は、後続の自動処理で勝手に上書きされてはならない。
3. **動画は外に出さない** — 動画とその編集情報はユーザーの PC 内で完結する。クラウド送信を前提にしない。
4. **長く編集しても壊れない** — 長時間処理、クラッシュ、誤操作、再起動を経ても、ユーザーの作業が失われにくい。

## 2. 想定ユーザーと利用シナリオ

### 2.1 想定ユーザー

| 種別 | 特徴 | 期待 |
|------|------|------|
| 個人クリエイター | 配信・公開前の動画から特定領域を隠したい | 1 本の動画を一気に処理して書き出したい |
| 編集担当者 | 複数本のレビュー前段として使う | 大量の候補を高速に確認・修正したい |
| 機密扱いの動画を持つ人 | 動画を外部に出せない | 完全ローカルで処理が完結することが最優先 |

### 2.2 中心となる利用シナリオ

1. ユーザーが動画ファイルを開く。
2. AI による自動検出を実行し、隠す候補のマスクトラックが提案される。
3. ユーザーは時間軸上でトラックを確認し、必要に応じて位置・形・期間を修正する。
4. ユーザーは作業途中でプロジェクトを保存する。後日、続きから編集できる。
5. ユーザーは書き出し設定を選び、モザイク入りの動画ファイルを生成する。

### 2.3 やらないこと (Out of Scope)

- クラウド側での AI 推論や、動画の外部送信を伴うサービス連携。
- 一般的な動画編集機能 (カット編集、トランジション、字幕、BGM 編集など)。
- モザイク以外の高度なエフェクト編集 (色補正、ぼかし以外のフィルタ等)。
- マルチユーザーでの同時編集や、リモート共有編集。

## 3. ドメインモデル要件

実装言語に依存しない、最低限存在しなければならない概念です。

### 3.1 プロジェクト

- ユーザーの 1 つの作業セッションは「プロジェクト」として独立して保存・読み込みできなければならない。
- プロジェクトには、対象動画の参照、編集中のマスクトラック群、検出設定、書き出し設定が含まれていなければならない。
- プロジェクトは再起動・別 PC 移送に耐えるよう、外部依存しないファイルとして保存できなければならない。

### 3.2 マスクトラック (編集の最小単位)

Auto Mosaic は、フレーム単位で孤立した図形を扱う方式 **ではない**。同じ対象を時間方向に追跡できる **マスクトラック** が編集の中心でなければならない。

マスクトラックは少なくとも次を表現できなければならない:

- 一意な識別子と、ユーザーが見分けるためのラベル
- 時間軸上に並ぶ複数のキーフレーム
- 各キーフレームでの図形 (楕円、多角形など) と任意の不透明度・拡張幅
- そのトラックが「自動検出由来」なのか「手動由来」なのかの出自情報
- ユーザーがそのトラックを編集・ロック・非表示にしたかどうかの状態

### 3.3 キーフレーム

- ユーザーは任意のフレーム位置にキーフレームを追加・移動・複製・削除できなければならない。
- キーフレームには、自動検出で生成されたものか、ユーザーが手で作成・編集したものかの区別が残らなければならない。

### 3.4 セグメントと有効区間

- マスクトラックは、その時間軸上で「描画してよい区間」と「描画してはならない区間」を区別できなければならない。
- 書き出し時に、有効でない区間に対してモザイクを描いてはならない。
- 直近のキーフレームを単純に保持し続けるだけの素朴な実装であってはならない。検出された区間と、補間されている区間と、信頼度の低い区間を区別できる必要がある。

### 3.5 動画参照

- プロジェクトは動画ファイルの実体パスをそのまま保持しなければならない。
- 表示用に変換された URL や仮想パスを永続データとして保存してはならない。

## 4. 機能要件

### 4.1 プロジェクト管理

- 新規作成、保存、別名保存、読み込みができる。
- 起動時に直近のプロジェクトを開く動線がある。
- 自動保存があり、明示保存と区別できる。
- 保存していない変更がある状態で破壊的操作 (新規・他プロジェクトを開く・動画を差し替える等) をしようとしたとき、ユーザーに確認を求める。
- 過去バージョンで作成されたプロジェクトを、現行バージョンで開ける (または安全に変換できる) こと。

### 4.2 動画読み込み

- 一般的な動画コンテナ・コーデックを開くことができる。
- 解像度、フレームレート、総フレーム数を取得し、UI に表示できる。
- 動画の読み込みは長時間処理として扱い、進捗とキャンセルが可能でなければならない。

### 4.3 自動検出

- 動画全体に対する一括検出ができる。
- 現在フレームのみに対する単発検出ができる。
- ユーザーが指定した区間 (開始/終了マーカー) に限定した検出ができる。
- 検出処理は長時間処理として扱い、進捗・キャンセル・状態確認ができる。
- 検出に失敗した、あるいはモデルが利用できない場合、UI 上で原因がわかる形で停止する。標準出力の文字化けや黙ったクラッシュであってはならない。
- **手動編集を破壊してはならない**: 自動検出の結果でユーザーの手動編集トラックや手動キーフレームを置き換えてはならない。範囲検出時は、範囲外のキーフレームと手動キーフレームを保護する。

### 4.4 マスクトラック編集

- ユーザーは、自動生成されたトラックとは別に、新規トラックを手動で作成できる。
- ユーザーは、トラックを削除、複製、分割できる。
- ユーザーは、トラックをロック・非表示・再表示できる。
- ユーザーが手で編集した、あるいは作成したトラックは「手動意図あり」として記録され、以降の自動処理から保護される。

### 4.5 図形編集

- ユーザーはプレビューキャンバス上で、楕円の移動・拡大・縮小・回転に相当する操作ができる。
- ユーザーはプレビューキャンバス上で、多角形の頂点の追加・移動・削除ができる。
- 数値入力 (バウンディングボックス、座標、形状種別、不透明度、拡張幅など) によっても編集できる。

### 4.6 キーフレーム編集

- ユーザーは任意のフレーム位置でキーフレームを追加・複製・削除・移動できる。
- 中間フレームは、隣接キーフレームから補間されたプレビューを見ることができる (楕円の bbox / 回転、多角形の頂点列、不透明度、拡張幅などが対象)。
- 手動キーフレームと自動キーフレームは、UI 上で見分けがつかなければならない。

### 4.7 タイムライン

- 動画全体を見渡せるタイムラインがあり、トラックとキーフレームが時間軸上に視覚化される。
- ズームイン・ズームアウトができる。
- プレイヘッドを表示し、再生・任意位置へのシークができる。
- キーフレームの状態 (確定 / 補間 / 信頼度低 / 検出範囲外 など) は色や形で区別できる。

### 4.8 プレビュー

- 動画と現在フレームのマスクが同期表示される。
- ユーザーは再生・停止・コマ送り・コマ戻しを行える。
- 再生位置はタイムラインと双方向に同期する。

### 4.9 書き出し

- ユーザーは解像度プリセット (元解像度を含む)、ビットレート (自動および手動)、音声の扱い、モザイクの強さを指定して書き出せる。
- 書き出しは長時間処理として扱い、進捗・キャンセル・成功/失敗状態が UI で確認できる。
- 書き出し前に、危険なフレーム (キーフレーム間隔が異常に空いている、面積が急変する、信頼度が極端に低いなど) を検出してユーザーに警告できる。
- 既存ファイルに上書きする前に確認する。
- ハードウェアエンコーダが利用可能な環境ではそれを使えるが、利用できない環境でも CPU エンコードで必ず完走できなければならない。

### 4.10 ジョブ管理

- 検出、動画読み込み、書き出し、モデル取得などの長時間処理は、すべて「ジョブ」として一覧で確認できる。
- 各ジョブは、待機中・実行中・キャンセル中・キャンセル済み・完了・失敗・中断のいずれかの状態を持つ。
- ユーザーは実行中のジョブをキャンセルできる。
- 既に終了したジョブを誤って「キャンセル中」状態に巻き戻してはならない。

### 4.11 環境診断

- ユーザーまたは開発者は、現在の実行環境 (Python / FFmpeg / GPU / モデル) の状態を 1 アクションで確認できる。
- モデルファイルは「存在するか」だけでなく、内容として正しいか (途中で壊れた、実体ではなくリダイレクト HTML だった、サイズが不正、など) まで検証されなければならない。

## 5. 不変条件 (どの実装でも守らなければならない約束)

ここに書かれているものは、UI 都合で曲げてはならず、永続データを扱うレイヤで守られなければなりません。

1. **手動編集の保護** — 手動で作成・編集・ロックされたトラックや手動キーフレームは、自動検出によって置き換えられない。
2. **マスクトラック中心** — 編集と書き出しの単位は、孤立したフレームの図形ではなく、時間方向に連続するマスクトラックである。
3. **状態の単一情報源** — 編集と書き出しが参照する状態は、永続化される側 (ドメイン側) を正とする。UI の見せ方を正にしてはならない。
4. **描画区間の明示** — どの区間にモザイクを描くかは、最後のキーフレームを永遠に維持する素朴な実装ではなく、確定 / 補間 / 信頼度低 / 範囲外を区別したうえで決まる。
5. **ローカル経路の維持** — 永続データに保存される動画やファイルへの参照は、OS のローカル経路をそのまま保持する。表示用に変換された URL を永続化してはならない。
6. **JSON 出力面の純粋性** — 機械可読の出力面と、人間向け / 診断用の出力面は明確に分けなければならない。診断ログが機械可読出力に混入してはならない。
7. **モデルは内容で判断する** — モデルファイルの可否は「ファイルが存在するか」ではなく、内容の妥当性検証で判断する。
8. **CPU フォールバックは必須** — GPU / アクセラレータが壊れている、または存在しない環境でも、検出と書き出しは CPU だけで完走できなければならない。

## 6. 非機能要件

### 6.1 プライバシー / オフライン性

- 動画ファイルそのもの、編集中のマスクデータを、ユーザーの明示的な許可なく外部送信してはならない。
- ネットワーク接続が一切ない環境でも、起動・読み込み・編集・書き出しがすべて完了できる。
- モデルファイルの取得など、ネットワークを必要とする操作は明確に分離され、その操作だけがネットワークを必要とする状態でなければならない。

### 6.2 信頼性 / 復旧性

- アプリが異常終了しても、自動保存または明示保存された時点までは復旧できる。
- 長時間処理が途中でクラッシュしても、その失敗が他の状態 (プロジェクト本体、他のジョブ) を破壊してはならない。
- 書き出し中に処理が落ちても、元の動画ファイルとプロジェクトファイルは破壊されない。

### 6.3 性能 / 応答性

- 一般的な PC で、フル HD 程度・数分〜十数分程度の動画を扱えること。
- UI は、長時間処理中もフリーズせず、進捗とキャンセル操作を受け付けられること。
- タイムラインと動画再生は、ユーザーが「ずれている」と感じない範囲で同期していること。

### 6.4 ユーザビリティ

- 主要なメニューと表示は日本語で提供される。
- 重要なショートカットキー (再生・停止・コマ送り・キーフレーム追加・削除・保存・取り消し・やり直し・I/O マーカー設定など) が用意されている。
- ヘルプを 1 アクションで参照できる。

### 6.5 国際化 / 文字コード

- 日本語ファイル名・パスを破壊せずに扱える。
- 表示も入出力も、UTF-8 を前提として一貫していなければならない。

### 6.6 セキュリティ

- 外部から取得したファイル (モデルなど) は、取得直後に検証され、検証に通ってから初めて利用領域に配置される。
- 外部プロセスを起動する箇所では、ユーザー入力のパスや文字列がコマンド注入として使われないように扱う。

## 7. 制約事項

- 主要対象 OS は Windows 11 (64bit)。それ以外の OS は副次的な対象とする。
- ローカル環境のみで動作することが前提であり、サーバ常駐コンポーネントやクラウドサービスを必須にしない。
- 商用クラウド API への依存を新規に追加しない。

## 8. 完成時に提供されるべきもの

要件定義としての「完成図」は、以下がすべて満たされている状態を指します。

### 8.1 ユーザーから見た完成

- 動画を開いてから書き出すまでの基本フロー (開く → 検出 → 確認・修正 → 書き出し) が、説明書なしでも到達可能。
- 自動検出と手動編集が混在する作業を 1 時間以上続けても、手動編集が失われない。
- 異常終了からの復帰時に、ユーザーが「直前まで何をしていたか」を判断できる手がかりが残っている。
- 書き出し前に「ここは怪しい」と分かる仕組みがあり、書き出しに進むかをユーザーが判断できる。
- ネットワークを切断したまま、最初から最後まで実行できる。

### 8.2 運用 / 配布から見た完成

- インストール、起動、アンインストールの手順が、エンジニアでなくても辿れる形で提供されている。
- 必要な依存物 (実行ランタイム、AI モデル、動画処理エンジン) は配布物から自己解決でき、ユーザーに別個のセットアップを強制しない。
- 環境診断機能で、不足や破損があれば「何を直せばよいか」がユーザーに伝わる。

### 8.3 開発から見た完成

- ドメインの不変条件 (5 章) を破る変更は、自動テストもしくはレビューで検出できる。
- バックエンド・フロントエンドの責務境界が、片方の都合でもう片方を歪めない形で維持されている。
- 機能追加時に、「手動編集が壊れない」ことを毎回確認しなくても済むように、その保証が仕組み化されている。

## 9. 用語集

| 用語 | 定義 |
|------|------|
| マスクトラック | 同じ対象を時間軸上で追いかける、複数キーフレームで構成されるモザイク領域の単位 |
| キーフレーム | あるフレーム時点でのマスクの形・位置・属性を確定する点 |
| 自動検出 | AI モデルによりマスク候補を自動生成する処理 |
| 手動編集 | ユーザーの操作によって作成・変更されたトラック / キーフレーム |
| ジョブ | 検出・書き出し・モデル取得など、UI からキャンセル可能な長時間処理の単位 |
| プロジェクト | 1 つの動画に対するすべての編集状態をまとめた永続データ |
| 危険フレーム | 補間信頼度・面積変化・キーフレーム間隔から、自動マスクが外れている可能性が高いと推定されるフレーム |

## 10. この文書のメンテナンス指針

- 実装が変わってもこの文書はすぐには変えない。**完成像** が変わったときに更新する。
- ファイル名・関数名・schema バージョンなど、実装詳細は **意図して書かない**。
- 5 章「不変条件」を変えるときは、その変更がドメインの意味を変える変更であることを、レビューで明示する。
- 「現状はこうです」という記述を増やしたくなったら、それは [`../engineering/current-implementation.md`](../engineering/current-implementation.md) 側に書く。
